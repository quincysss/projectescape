# 《废土生存法则》多人模式预留架构（修订版）

> 文档版本：v0.1  
> 所属项目：《废土生存法则》  
> 前置文档：01_项目总纲.md / 02_核心循环规则.md / 05_容器刷新与开箱系统.md / 06_前哨材料与前哨站修复系统.md / 08_撤离与结算系统.md  
> 适用阶段：V0.1 起预留，V0.5 之后预研  
> 文档目标：定义单人版本开发时必须保留的多人扩展边界，避免后续重构过大，同时明确早期不做真实联机。

---

## 1. 本文档一句话说明

V0.1 到 V0.4 不做真实多人，但所有关键局内行为都应以“未来可服务器校验”的方式组织。

---

## 2. 多人模式目标

后续可能方向：

```text
双人合作搜刮撤离。
多人各自为战抢资源。
1v1v1 小局资源竞争。
非对称对抗。
观战或幽灵干扰模式。
```

当前不决定最终形态。

当前只预留：

```text
数据归属。
命令式交互。
对象唯一 ID。
可重放事件。
结算记录。
```

---

## 3. 早期不做什么

V0.1 到 V0.4 不做：

```text
真实网络同步。
房间匹配。
服务器权威战斗。
PVP 伤害。
多人聊天。
好友系统。
账号系统。
掉线重连。
```

原因：

```text
核心玩法还没有完全验证。
过早联机会放大所有系统成本。
单人搜刮撤离成立后，多人模式才值得投入。
```

---

## 4. 架构预留原则

## 4.1 Command 优先

关键行为不要让 UI 直接改数据。

推荐都走命令：

```text
InteractContainerCommand
OpenContainerCommand
TakeItemCommand
StoreToSafeCommand
SubmitOutpostMaterialCommand
StartExtractCommand
ConfirmSettlementCommand
EquipItemCommand
UseConsumableCommand
```

命令字段：

```text
command_id
player_id
run_id
target_id
payload
client_time
sequence
```

单人版本：

```text
本地 CommandBus 直接执行。
```

多人版本：

```text
客户端发送命令。
服务器校验。
服务器广播结果事件。
```

---

## 4.2 对象 ID 稳定

局内对象必须有唯一 ID：

```text
run_id
player_id
container_id
spawn_point_id
outpost_id
item_instance_id
equipment_instance_id
safe_storage_id
extract_zone_id
```

规则：

```text
不要依赖节点路径作为逻辑 ID。
不要依赖数组下标作为持久 ID。
生成对象时立即分配 ID。
存档、结算、日志都使用 ID。
```

---

## 4.3 权威归属字段

多人预留字段：

```text
owner_player_id
last_interactor_id
locked_by_player_id
reserved_until_time
claim_policy
visibility_policy
```

用于：

```text
容器读条归属。
前哨材料提交归属。
安全存储归属。
撤离成功归属。
局内拾取物归属。
```

V0.1 单人可以默认：

```text
player_id = local_player
owner_player_id = local_player
```

---

## 5. 系统级多人预留

## 5.1 容器系统

预留：

```text
container_id
state
opened_by_player_id
looted_by_player_ids
lock_policy
remaining_items
despawn_time
```

多人策略可选：

```text
共享容器：所有玩家看到同一个物资池。
独占容器：第一个打开者拥有短暂优先权。
队伍容器：队伍共享，外人不可拿。
```

建议后续优先：

```text
共享容器 + 短暂交互锁。
```

---

## 5.2 前哨系统

预留：

```text
outpost_id
repair_progress
contributor_records
team_shared
ownership_policy
```

多人策略可选：

```text
合作共享修复。
各玩家独立修复。
阵营抢占修复。
```

建议：

```text
先支持合作共享修复。
贡献记录只用于结算和奖励展示。
```

---

## 5.3 背包与安全存储

预留：

```text
inventory_owner_id
safe_storage_owner_id
item_lock_state
transaction_id
```

规则：

```text
玩家背包永远私有。
安全存储默认私有。
多人合作版本可增加队伍共享箱，但不替代个人安全存储。
```

---

## 5.4 撤离与结算

预留：

```text
extract_attempt_id
extract_zone_id
extract_player_ids
extract_start_time
extract_result
settlement_records
```

多人撤离策略：

```text
个人撤离：每个玩家独立成功或失败。
队伍撤离：队伍成员都在范围内才成功。
抢点撤离：撤离点短时间内只能服务一个玩家或队伍。
```

建议：

```text
优先做个人撤离。
合作模式再做队伍撤离。
```

---

## 6. 时间与暂停规则

多人预留要求：

```text
局内时间不暂停。
容器界面不暂停。
背包界面不暂停。
结算前所有局内倒计时继续。
```

原因：

```text
多人无法为一个玩家暂停世界。
单人早期保持不暂停，可以减少后续返工。
```

已确定规则：

```text
打开容器 UI 时玩家锁移动，但世界时间继续。
容器倒计时继续。
撤离读条期间对局倒计时为 0 则撤离失败。
```

## 7. 网络模型预研

未来可选：

```text
本地联机。
Steam P2P。
专用服务器。
房主权威。
服务器权威。
```

建议路线：

```text
V0.5 原型：房主权威。
V0.6 如果 PVP 成立，再评估服务器权威。
```

原因：

```text
游戏早期重点是验证多人乐趣，不是先解决所有反作弊问题。
如果未来要做强 PVP，再升级权威模型。
```

```text

ItemTakenEvent
OutpostSubmittedEvent
OutpostRepairedEvent
EquipmentChangedEvent
ConsumableUsedEvent
ExtractStartedEvent
ExtractCompletedEvent

```

用途：

```text
调试。
回放。
结算。

异常排查。
```

V0.1 可只记录调试日志。

## 9. Godot 实现建议

推荐模块：

```text
CommandBus
RunEventBus
RunObjectRegistry
PlayerSessionData
NetworkFacade
AuthorityService
PredictionStub
ReplicationStub
```

说明：

```text
NetworkFacade 在单人版本为空实现。
AuthorityService 在单人版本默认本地通过。
PredictionStub 和 ReplicationStub 先只留接口。
```

关键原则：

```text
业务系统调用 CommandBus，不直接调用网络 API。
网络层只负责传输命令和事件。
系统逻辑仍在各自 Manager 中。
```

## 10. 验证清单

```text
关键交互都有 Command 名称。
局内对象都有稳定 ID。
容器、前哨、背包、撤离都保存 player_id。
UI 不直接修改核心数据。
打开 UI 不暂停局内时间。
结算记录能区分玩家。
单人版本不需要真实网络也能运行。
```

---

## 11. 当前设计体检与建议

合理点：

```text
先做单人，避免被联机复杂度拖死。
Command 预留能降低后续重构成本。
不暂停规则已经和多人兼容。
容器和前哨天然适合多人争夺或合作。
```

风险：

```text
如果后续要强 PVP，当前数值和地图需要重新评估。
如果物品归属规则不早定，多人会出现大量争议。
如果 UI 直接改数据，后续网络化会很痛苦。
```

建议：

```text
现在只做接口和字段预留。
不要为了未来多人牺牲当前单人手感。
所有多人相关代码都允许是空实现。
```

---

## 12. 本文档结论

多人模式现在不该进入制作阶段。

当前最重要的是：

```text
Command 化交互。
稳定对象 ID。
玩家归属字段。
不暂停局内时间。
结算事件可追踪。
```

做到这些，后续无论走合作、竞争还是非对称，都不会推翻整个单人框架。

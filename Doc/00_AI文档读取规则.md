# AI 文档读取规则

> 目标：把 `Doc/` 从“全量阅读资料库”变成“按任务路由的真相源”。

## 1. 权威级别

| 级别 | 类型 | 使用方式 |
|---|---|---|
| L0 | 当前用户消息 | 最高优先级。用户当轮纠正必须同步进文档。 |
| L1 | 已稳定运行的工程事实、数据表、测试结果 | 可以反写执行文档，但不要直接覆盖长期设计意图。 |
| L2 | 系统 `00_AI开工入口.md`、拆分执行文档、验收清单 | 实现和文档维护的主要依据。 |
| L3 | 系统设计母版 | 解释目标、体验意图、长期边界。 |
| L4 | 数据、美术、UI、音频、世界观等专题规范 | 跨系统约束，按任务读取。 |
| L5 | 旧稿、历史说明、归档候选 | 只作溯源，不直接指导实现。 |

冲突处理顺序：

```text
当前用户消息
已稳定工程事实
系统 AI 开工入口
拆分执行文档
设计母版
专题规范
旧稿/历史说明
```

## 2. 文档类型

| 类型 | 作用 | 修改规则 |
|---|---|---|
| AI 开工入口 | 告诉 AI 该读什么、怎么验收 | 只写路由、必读文档、验收入口，不写完整玩法规则 |
| 系统索引 | 从任务关键词路由到系统和权威文档 | 新增/移动/归档文档时必须同步 |
| BDD 需求规范 | 约束需求、验收、AI 提示词的行为场景写法 | 后续需求文档整理必须使用 `Feature / Scenario / Given / When / Then` |
| 设计母版 | 记录系统目标、体验意图、长期边界 | 只改长期规则，不写临时工程绕路 |
| 拆分执行文档 | 记录可落地路径、接口、字段、验收 | 实现任务优先修改这里 |
| 验收清单 | 证明任务完成的标准 | 每次规则变化后检查是否仍覆盖 |
| AI 开工提示词 | 给后续 AI/MCP 执行的压缩任务说明 | 必须引用具体路径、接口和验收项 |
| 工程维护文档 | 记录重构、结构调整、风险 | 不与玩法母版混写 |

## 3. 新增文档规则

新增文档前先判断它属于哪一类。新增后必须同步：

1. `Doc/系统文档索引表.md`
2. 目标系统 `00_AI开工入口.md`
3. `Doc/README.md`
4. `.codex/skills/projectescape-production/references/doc-map.md`
5. 如影响未来执行纪律，同步 `.codex/skills/projectescape-production/SKILL.md`

新增文档必须写清：

```text
适用范围：
权威级别：
上游依赖：
下游影响：
不负责内容：
BDD 场景：
验收入口：
```

如果新增文档属于需求、功能规则、验收清单或 AI 开工提示词，必须按 `Doc/00_BDD需求文档规范.md` 增加至少一个 `Feature` 和一个 `Scenario`。

## 4. 修改文档规则

修改前先回答：

```text
这次改的是长期设计，还是工程落地？
是否属于需求文档整理，是否需要 BDD 场景？
会不会影响数据表、代码路径、资源路径、AI 开工提示词？
是否存在同一规则的其他旧口径？
```

修改后必须：

- 搜索旧口径残留，例如旧数值、旧字段名、旧路径、旧系统名。
- 检查系统入口是否需要更新。
- 检查索引表是否需要更新。
- 检查验收清单是否需要新增场景。
- 检查 BDD 场景是否覆盖成功、失败和边界条件。
- 最终汇报说明未跑自动化验证的原因。

## 5. 删除与归档规则

默认先归档，不直接删除设计文档。

删除或归档前必须记录：

```text
删除/归档原因：
替代文档：
是否影响 AI 开工入口：
是否影响 skill/doc-map：
是否仍被代码、表格、资源或提示词引用：
```

允许直接删除的通常只有：

- 明确重复且内容已被新文档完整吸收的临时稿。
- 已确认不再使用的临时提示词。
- 与工程事实冲突且无设计价值的过期执行说明。

## 6. UTF-8 与验证

- 中文 `.md`、`.gd`、`.tscn`、`.tab`、skill 文件必须按 UTF-8 读写。
- Windows 下读取中文文件时显式使用 UTF-8，避免把正常中文误判为乱码。
- 文档任务至少做文本验证：目录存在、入口存在、旧口径搜索、索引同步。
- 代码任务按相关系统验收清单执行最窄测试；没有自动化测试时说明手动检查范围。

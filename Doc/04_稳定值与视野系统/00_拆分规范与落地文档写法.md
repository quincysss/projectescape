# 00_拆分规范与落地文档写法

> 来源文档：`../04_稳定值与视野系统_修订版_废土生存法则.md`  
> 参考规范：`../02_核心循环规则/00_拆分规范与落地文档写法.md`  
> 目标：把稳定值与视野系统拆成可直接实现的 Godot 组件、配置、反馈、死亡、调试与 AI/MCP 开工文档。

---

## 1. 拆分原则

04 是系统实现文档，不应只拆成概念说明。每份落地文档都要能让执行代理明确知道：

```text
要新增哪个组件
组件持有哪些字段
每帧或每次事件如何计算
信号如何发出
哪些系统能调用它
如何调试和验收
```

每份文档必须满足：

| 要求 | 说明 |
|---|---|
| 可编码 | 能直接创建 Godot 脚本或资源 |
| 可调参 | 核心数值进入配置，不写死在逻辑里 |
| 可联动 | 明确和 02 核心循环、03 安全区、05 容器、06 前哨、07 背包的接口 |
| 可验证 | 有手动/自动调试入口，能验证阶段、半径、遮罩、死亡 |
| 可喂 AI | 每个模块有独立可复制的 AI/MCP 开工提示词 |

---

## 2. 推荐文档结构

```text
# 文档名

## 1. 实现目标
本模块最终要在 Godot 中产生什么能力。

## 2. 设计依据
来自母版文档的关键规则和数值。

## 3. 功能规则细化
触发、前置、执行、中断、失败、反馈、边界。

## 4. Godot 落地范围
建议新增/修改 scripts、scenes、resources、autoload。

## 5. 数据结构
字段、枚举、配置、信号参数。

## 6. 核心流程
每帧计算、事件响应、状态转换或伪代码。

## 7. 信号与接口
对外发什么、监听什么、谁调用谁。

## 8. AI/MCP 开工提示词
可直接复制给执行代理。

## 9. 验收标准
可验证的行为结果。

## 10. 暂不实现
防止扩散到后续版本或其他系统。
```

| 类型 | 建议路径 |
|---|---|
| 稳定值组件 | `res://scripts/player/StabilityComponent.gd` |
| 稳定值配置 | `res://resources/stability/StabilityConfig.tres` |

| UI | `res://scripts/ui/StabilityHUD.gd` |
| 音频反馈 | `res://scripts/audio/StabilityAudioController.gd` |
| 死亡控制 | `res://scripts/player/PlayerDeathController.gd` |
| 调试面板 | `res://scripts/debug/StabilityDebugPanel.gd` |

## 5. 核心命名约定

内部命名统一使用：

```text
stability
current_stability
max_stability
stability_stage
vision_radius
target_vision_radius
actual_vision_radius
darkness_alpha
area_decay_multiplier
equipment_decay_multiplier
item_decay_multiplier
```

稳定值阶段：

```gdscript
enum StabilityStage {
    STABLE,
    UNSTEADY,
    DANGER,
    CRITICAL,
    DEPLETED
}
```

## 6. AI/MCP 提示词质量底线

每份提示词必须包含：

```text
任务目标
必须读取的文档
允许新增/修改范围
禁止实现范围
完成标准
验证方式
最终报告格式
```

禁止模糊提示：

```text
把稳定值做好
优化视野效果
按文档实现
加点紧张感
```

合格提示示例：

```text
请实现 StabilityComponent。
读取 01_稳定值核心组件.md 和 13 对应小节。
新增 StabilityComponent.gd 与 StabilityConfig.tres。
实现危险区下降、安全区回复、倍率修正、阶段变化、归零信号。
禁止实现 UI、音频、死亡演出和物品效果。
完成后用 DebugStabilityVision 测试：离开安全区下降，进入家回复，进入前哨回复，归零只发一次 depleted。
```

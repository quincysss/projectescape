# 32-03 开店中营业阶段规则

> 开店中营业阶段对应 `SHOP_OPEN`。这是 60 秒限时经营阶段，玩家主要围绕货台、补货、撤货和需求榜做即时决策。

## 1. 阶段目标

开店中要解决的问题：

```text
有限货台格子摆什么
热销货卖完后是否及时补货
低需求货是否撤下换高需求货
没有货时是否提前结束
```

## 2. 界面结构

建议布局：

```text
顶部：第 N 天 / 开店中 / 剩余时间 60s -> 0s / 当前矿币

左侧：货台区
- 初始 3 格
- 每格显示图标、名称、数量、预计成交状态

中间：可上架物资列表
- 只显示仓库中的 sale_good
- 显示库存数量
- 支持点击上架或补货

右侧：今日需求榜
- 排名、热度、售价修正、预计成交间隔

底部：成交日志
- 最近成交、难民离开、缺货提示

右下：提前结束营业按钮
```

## 3. 货台规则

### 3.0 货台所有权规则

货台不是独立仓库。V0.2 推荐实现为：

```text
仓库物品实例仍归 WarehouseManager 管理
上架只是把对应数量标记为 ON_SHELF
ShelfInventoryService 负责读取和改写货台状态
```

这样可以避免营业结束回仓时因为仓库容量变化吞物。

货台上的每一格应记录：

```text
shelf_slot_id
unlocked
item_id
quality
quantity
source_stack_id
state
next_sale_second
last_sale_second
```

其中 `source_stack_id` 用于追踪货台数量来自哪个仓库堆叠；如果工程暂时没有堆叠实例 ID，可先使用 `item_id + quality` 聚合，但必须保留未来扩展字段。

### 3.1 货台格子数

```text
初始货台格子数 = 3
后续可由研究所提升
```

格子状态：

| 状态 | 显示 | 可操作 |
|---|---|---|
| 空 | 空货台 | 选择上架 |
| 有货 | 图标、名称、数量 | 补货、撤下、详情 |
| 售空 | 售空提示 | 补货或保持空 |
| 锁定 | 锁链/暗色 | 不可操作 |

### 3.2 上架

上架条件：

- 物品必须是 `item_type = sale_good` 或 tags 包含 `sale_good`。
- 物品必须在仓库中且未被研究、制造、出发准备占用。
- 上架数量不能超过仓库可用数量。

上架后：

```text
仓库对应数量转为 ON_SHELF 状态
货台显示该物资数量
该数量不可被研究、制造或出发准备消耗
```

上架流程：

```text
1. 玩家点击空货台。
2. 系统打开可上架物资选择列表。
3. 列表只显示 AVAILABLE 且 sale_good=true 的物资。
4. 玩家选择物资。
5. 玩家选择上架数量，默认值为 min(库存可用数量, 推荐上架数量)。
6. ShelfInventoryService 执行上架。
7. ShopSalesService 为该货台生成 next_sale_second。
```

推荐上架数量：

| 情况 | 推荐数量 |
|---|---:|
| 今日需求榜第 1 名 | min(库存, 3) |
| 今日需求榜第 2-3 名 | min(库存, 2) |
| 其他上榜物资 | min(库存, 1) |
| 未上榜物资 | 1 |

玩家可以手动调整数量，但不能小于 1，也不能超过可用库存。

### 3.3 补货

补货条件：

- 货台已有同一种 `sale_good`。
- 仓库中仍有相同 item_id、quality 和状态可合并的库存。
- 补货数量不超过单格货台上限。

V0.2 可先使用固定单格货台上限：

```text
每个货台格最多展示 99 个同种可售物资
```

补货流程：

```text
1. 玩家点击已有货台。
2. 系统显示补货、撤下、查看详情。
3. 玩家选择补货。
4. 系统显示可补数量。
5. 玩家确认后，ShelfInventoryService 把仓库 AVAILABLE 数量转为 ON_SHELF。
6. 如果该货台原本售空，ShopSalesService 重新生成 next_sale_second。
```

如果货台仍有货，补货不应重置已经临近的成交计时，避免玩家通过反复补货刷新随机时间。只有物资从 0 变为大于 0 时才重新生成 `next_sale_second`。

### 3.4 撤下

玩家可以撤下货台中的未售出物资。

撤下后：

```text
货台数量减少
物资回到仓库 AVAILABLE 状态
成交计时重新计算
```

撤下不能回滚已经成交的物资。

撤下规则：

| 操作 | 结果 |
|---|---|
| 撤下全部 | 货台变为空，清空 `next_sale_second` |
| 撤下部分 | 货台保留剩余数量，成交计时继续 |
| 售空状态撤下 | 无可撤下物资，只能补货或保持空 |
| 已成交记录 | 不受撤下影响 |

如果玩家在成交触发同一帧撤下物资，优先处理已经到点的成交，再处理玩家撤下输入，避免出现负数或成交记录与数量不一致。

### 3.5 查看详情

玩家可在 `SHOP_OPEN` 中查看货台物资 Tooltip。

Tooltip 至少显示：

```text
物资名
品质
今日需求排名
当前售价修正
预计成交间隔
货台数量
仓库可补数量
基础描述
```

Tooltip 只读，不允许在 Tooltip 内直接出售、制造或研究。

## 4. 自动成交规则

每个货台格独立计算成交。成交模拟由 `ShopSalesService` 按 `shop_elapsed_seconds` 推进，不由 UI 动画或 Label 文本驱动。

```text
rank = 今日需求榜中该物资排名
base_interval = rank 对应成交间隔
next_sale_time = 当前营业秒 + 随机间隔
```

建议默认值：

| 需求排名 | 成交间隔 | 售价修正 |
|---|---:|---:|
| 第 1 名 | 4-7 秒 | 1.25x |
| 第 2-3 名 | 7-12 秒 | 1.10x |
| 第 4-6 名 | 12-20 秒 | 1.00x |
| 第 7 名以后 | 20-35 秒 | 0.85x |
| 未上榜 | 35-60 秒或不成交 | 0.70x |

### 4.1 成交调度

生成 `next_sale_second` 时必须满足：

```text
next_sale_second > shop_elapsed_seconds
next_sale_second <= shop_duration_seconds
```

如果随机出的下一次成交时间超过营业结束时间：

- 该货台可以显示“本日可能来不及卖出”。
- 不再强行成交。

如果同一秒多个货台同时到达成交时间，按以下顺序处理：

```text
需求排名更高者优先
排名相同则货台 slot_index 更小者优先
仍相同则按 item_id 字典序，保证可复现
```

### 4.2 成交价格

实际单价建议：

```text
actual_unit_price = round(base_sale_value * price_multiplier)
```

若 `base_sale_value <= 0`：

- 使用 `items.tab` 或 `sale_good_recipes.tab` 中的兜底售价。
- 仍为 0 时，本次成交不应崩溃，但成交日志标记为 `price_missing`，并按 0 收入处理。

成交成功：

```text
货台数量 -1
记录成交：item_id、name、quality、unit_price、quantity=1、demand_rank、time
本日预结算收入增加
刷新该货台下一次成交时间
```

如果货台售空：

```text
该格变为售空或空
停止该格成交计时
提示“货台售空”
```

### 4.3 未满足需求

V0.2 不需要完整 NPC 排队，但需要记录“未满足需求”的轻量事件。

当当前时间触发一次难民需求检查，但货台没有匹配商品时：

```text
missed_customer_count +1
成交日志显示“有人想买 {需求标签或物资}，但货台没有对应货品”
不扣货
不加币
```

未满足需求可由 `ShopSalesService` 根据今日需求榜定期生成，也可在没有任何可成交货台时按低频率生成。它用于让玩家知道摆错货会浪费营业时间。

### 4.4 暂停与恢复

以下情况应暂停成交推进：

- 打开系统级暂停菜单。
- 打开营业结算面板。
- 执行提前结束确认弹窗时，若工程统一冻结局外输入。
- Debug 暂停。

以下情况不暂停成交推进：

- 查看 Tooltip。
- 浏览需求榜。
- 打开货台补货选择弹窗，除非工程明确使用模态暂停。

如果选择弹窗不暂停营业，UI 必须显示倒计时仍在减少。

## 5. 开店中禁止行为

`SHOP_OPEN` 中禁止：

- 开始研究。
- 制造新物资。
- 同类别材料转化。
- 整理仓库排序。
- 出发探索。
- 进入夜间计划。
- 打开旧独立买卖界面。

允许：

- 查看仓库可上架物资列表。
- 货台上架、补货、撤下。
- 查看 Tooltip。
- 查看需求榜。
- 查看成交日志。
- 点击提前结束营业。

## 6. 没货状态

以下条件同时满足时，系统判定玩家“当前没货可卖”：

```text
货台所有可售物资数量 = 0
仓库中可上架 sale_good 数量 = 0
```

没货后：

- 成交模拟暂停。
- 成交日志显示“货台已经没有可售物资”。
- 提前结束营业按钮高亮。
- 不强制等待剩余倒计时结束。

如果玩家之后通过某种合法方式又获得可上架 `sale_good`，可取消没货提示。但 V0.2 中 `SHOP_OPEN` 不允许制造，因此通常只能提前结束。

没货状态不等于自动结算。系统只高亮提前结束按钮，最终是否提前结束由玩家确认，除非 Debug 或剧情脚本显式触发。

## 7. 异常与边界

| 情况 | 处理 |
|---|---|
| 货台数量变成负数 | 立即修正为 0，记录 ERROR，停止该格成交 |
| 上架物品配置被移除 | 货台显示缺失配置，占位并禁止成交 |
| 读档时 `next_sale_second` 小于当前时间 | 立即重新生成下一次成交时间 |
| 读档时货台引用的仓库堆叠不存在 | 放入 `pending_return_shelf_items`，禁止静默销毁 |
| 货台锁定但有物资 | 允许结算回仓，不允许继续成交 |

## 8. BDD 场景

Feature: 开店中营业阶段

  Scenario: 玩家上架可售物资
    Given 当前阶段为 SHOP_OPEN
    And 仓库中有 3 个 sale_good_simple_gear
    When 玩家点击空货台并选择上架 2 个 sale_good_simple_gear
    Then 货台应显示 sale_good_simple_gear x2
    And 仓库中 sale_good_simple_gear 可用数量应减少 2

  Scenario: 热门货物自动成交
    Given 当前阶段为 SHOP_OPEN
    And 今日需求榜第 1 名为 sale_good_simple_gear
    And 货台上有 sale_good_simple_gear x2
    When 营业时间推进到该货台的 next_sale_time
    Then sale_good_simple_gear 货台数量应减少 1
    And 本日成交记录应新增 sale_good_simple_gear x1

  Scenario: 没货时高亮提前结束
    Given 当前阶段为 SHOP_OPEN
    And 所有货台数量均为 0
    And 仓库中没有可上架 sale_good
    When 系统刷新营业界面
    Then 界面应显示“当前没有可售物资”
    And 提前结束营业按钮应高亮
    And 系统不应强迫玩家等待倒计时归零

  Scenario: 补货不刷新已有成交倒计时
    Given 当前阶段为 SHOP_OPEN
    And 货台 0 上有 sale_good_simple_gear x1
    And 货台 0 的 next_sale_second 为 20
    When 玩家在第 15 秒给货台 0 补货 2 个 sale_good_simple_gear
    Then 货台 0 的数量应变为 3
    And 货台 0 的 next_sale_second 仍应为 20

  Scenario: 未上榜货物低频成交
    Given 当前阶段为 SHOP_OPEN
    And sale_good_low_value 未进入今日需求榜
    And 货台上有 sale_good_low_value x1
    When 系统生成该货台成交时间
    Then next_sale_second 应使用未上榜成交区间
    And 成交单价应使用未上榜售价修正

  Scenario: 没有匹配货物时记录未满足需求
    Given 当前阶段为 SHOP_OPEN
    And 今日需求榜第 1 名为 sale_good_water_filter
    And 所有货台都没有 sale_good_water_filter
    When 系统触发一次 sale_good_water_filter 需求检查
    Then missed_customer_count 应增加 1
    And 成交日志应显示未满足需求提示

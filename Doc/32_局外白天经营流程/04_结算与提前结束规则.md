# 32-04 营业结算与提前结束规则

> 本文专门定义“开店期间没货了怎么办”。结论：玩家必须可以提前结束营业，立即进入本日开店结算，再进入夜晚流程。

## 1. 进入结算的来源

`SHOP_SETTLEMENT` 有两个来源：

| 来源 | 触发条件 | 结算类型 |
|---|---|---|
| 自然结束 | SHOP_OPEN 倒计时到 0 | 完整营业结算 |
| 提前结束 | 玩家点击提前结束营业 | 提前结束结算 |

两种结算使用同一套数据结构和面板，只在标题或备注中区分是否提前结束。

进入 `SHOP_SETTLEMENT` 前必须生成一次 `ShopSettlementSnapshot`。

推荐字段：

```text
shop_session_id
shop_day
ended_by
elapsed_seconds
duration_seconds
sales_records
sold_summary
missed_customer_count
total_earned
unsold_returned_goods
pending_return_shelf_items
settlement_applied
created_at
```

`ShopSettlementSnapshot` 是结算面板和读档恢复的唯一展示来源。结算面板不得重新扫描货台来推导收益，避免读档或回仓状态变化导致数字漂移。

## 2. 提前结束按钮

### 2.1 显示条件

`SHOP_OPEN` 全程显示提前结束营业按钮。

按钮文案：

```text
提前结束营业
```

### 2.2 普通状态

当还有货可卖时：

- 按钮正常显示。
- 点击后弹出二次确认。
- 确认文案：`营业时间还没结束，是否提前结算今天的收益？`

### 2.3 没货高亮状态

当满足“货台为空且仓库无可上架 sale_good”：

- 按钮高亮。
- 可显示提示：`没有可售物资了，可以提前结束。`
- 点击后仍建议二次确认，但默认按钮聚焦在“确认结算”。

## 3. 提前结束执行流程

玩家确认提前结束后：

```text
1. 停止 SHOP_OPEN 倒计时。
2. 停止所有货台成交计时器。
3. 冻结当前成交记录。
4. 将未售出的货台物资回仓或标记待回仓。
5. 根据本日成交记录计算收益。
6. 通过 CurrencyWallet 增加矿币。
7. 写入 shop_settlement_applied = true，防止重复加币。
8. 切换 outgame_phase = SHOP_SETTLEMENT。
9. 显示今日营业结算面板。
```

注意：成交发生时可以先记入本日临时收入，但正式加币建议在结算应用时统一完成。若工程选择成交即时加币，则结算必须只展示结果，不得重复加币。

## 3.1 自然结束执行流程

倒计时自然归零时，执行与提前结束相同的结算管线，只是：

```text
shop_ended_by = timer_end
标题显示“今日营业结算”
不显示提前结束备注
```

自然结束与提前结束不得维护两份独立逻辑。任何回仓、加币、防重复、成交汇总和进入夜晚规则都必须复用同一套函数或服务。

## 3.2 结算应用时机

推荐结算应用顺序：

```text
1. build_settlement_snapshot()
2. return_unsold_goods()
3. apply_currency_once()
4. mark_shop_settlement_applied()
5. save_game()
6. show_settlement_panel()
```

如果第 2 或第 3 步失败，不应进入夜晚。

失败处理：

| 失败点 | 处理 |
|---|---|
| 回仓失败 | 保留 `pending_return_shelf_items`，阻止进入夜晚 |
| 加币失败 | 保持 `shop_settlement_applied=false`，允许重试 |
| 保存失败 | 提示保存失败，保留结算面板，不重复执行加币 |
| Snapshot 构建失败 | 停留在 `SHOP_OPEN` 暂停状态，并提示错误 |

## 4. 未售出物资处理

结算时，货台中未售出的物资必须回仓。

```text
ON_SHELF -> AVAILABLE
```

如果回仓失败：

- 不允许静默吞物。
- 保持在 `pending_return_shelf_items`。
- 结算面板显示“有未回仓物资待处理”。
- 不进入夜晚前必须处理完成，或保存 pending 状态供读档恢复。

V0.2 推荐：货台物资本质仍在仓库，只是状态为 `ON_SHELF`，因此结算回仓只需改状态，避免容量不足导致回仓失败。

未售出物资回仓不产生新的仓库入库动画，也不触发图鉴新增提示。它只是状态恢复。

若玩家提前结束时仍有货台商品：

```text
已成交数量不返还
未成交数量回仓
该商品的成交计时清空
```

如果同一个 `item_id` 在多个货台存在，结算汇总时可以聚合显示，但回仓时必须逐格恢复对应 `source_stack_id` 或聚合堆叠。

## 5. 结算面板内容

标题：

```text
今日营业结算
```

如果提前结束：

```text
今日营业结算（提前结束）
```

列表字段：

| 字段 | 说明 |
|---|---|
| 物资图标 | sale_good 图标 |
| 物资名 | 例如连轴齿轮 |
| 卖出数量 | 本日成交数量 |
| 单价 | 实际成交单价，可取平均价 |
| 小计 | 该物资总收益 |

底部居中：

```text
总收入：{total_earned} 矿币
```

如果一件都没卖出：

```text
今天没有卖出任何物资。
总收入：0 矿币
```

建议增加：

| 字段 | 说明 |
|---|---|
| 营业时长 | 已营业秒数 / 总秒数 |
| 未满足需求人数 | `missed_customer_count` |
| 未售回仓 | 展示回仓物资种类和数量，可折叠 |
| 结束原因 | 自然结束、提前结束、Debug 结束 |

结算面板只允许一个主按钮：

```text
进入夜晚
```

如果存在 `pending_return_shelf_items`，按钮置灰，提示“仍有未回仓物资待处理”。

按钮：

```text
进入夜晚
```

点击后：

```text
outgame_phase = NIGHT
```

进入夜晚时还需要：

```text
清空当前 shop_session 运行中计时器
保留结算历史摘要
清空货台临时成交调度
保留已更新后的 CurrencyWallet
保存 outgame_phase=NIGHT
```

## 6. 读档与防重复结算

必须保存：

- `shop_session_id`
- `shop_started_at_day`
- `shop_elapsed_seconds`
- `shop_ended_by`
- `shop_sales_records`
- `shop_total_earned`
- `shop_settlement_applied`
- `shelf_items`
- `pending_return_shelf_items`

规则：

```text
shop_settlement_applied = false
  可以应用加币和回仓。

shop_settlement_applied = true
  只能展示结算结果，不得再次加币。
```

如果游戏在 `SHOP_OPEN` 退出：

- 读档恢复到 `SHOP_OPEN`。
- 保留剩余时间、货台和成交记录。
- 玩家仍可继续营业或提前结束。

如果游戏在 `SHOP_SETTLEMENT` 退出：

- 读档恢复结算面板。
- 不重复加币。
- 关闭后进入 `NIGHT`。

### 6.1 防重复加币规则

`shop_settlement_applied` 是防重入主字段。

```text
false：允许执行 apply_currency_once()
true：禁止再次加币，只能展示 snapshot
```

`CurrencyWallet` 也建议记录最近一次应用的 `shop_session_id`：

```text
last_applied_shop_session_id
```

当 `shop_settlement_applied=false` 但 `CurrencyWallet.last_applied_shop_session_id == shop_session_id` 时，说明存档中存在不一致，系统应：

- 不再次加币。
- 修正 `shop_settlement_applied=true`。
- 记录 WARN，提示结算状态已修复。

### 6.2 SHOP_OPEN 读档恢复

读档恢复 `SHOP_OPEN` 时：

```text
剩余时间 = shop_duration_seconds - shop_elapsed_seconds
恢复 shelf_items
恢复 shop_sales_records
恢复 shop_total_earned 临时账本
恢复每个货台的 next_sale_second
```

若 `shop_elapsed_seconds >= shop_duration_seconds`，读档后应立刻进入结算管线，不应回到可操作营业界面。

## 7. 结算历史

每次成功关闭结算进入夜晚后，建议写入一条简短经营历史：

```text
shop_day
shop_session_id
ended_by
total_earned
sold_item_count
missed_customer_count
top_sold_item_id
```

经营历史用于 Debug、后续经营日志和难民需求反复降权，不作为玩家核心背包数据。

## 8. BDD 场景

Feature: 营业提前结束与结算

  Scenario: 没货后提前结束营业
    Given 当前阶段为 SHOP_OPEN
    And 开店剩余时间为 35 秒
    And 所有货台数量均为 0
    And 仓库中没有可上架 sale_good
    When 玩家点击提前结束营业并确认
    Then 当前阶段应变为 SHOP_SETTLEMENT
    And 系统应显示今日营业结算面板
    And 玩家不需要等待剩余 35 秒

  Scenario: 有货时提前结束需要确认
    Given 当前阶段为 SHOP_OPEN
    And 货台上有 sale_good_simple_gear x2
    When 玩家点击提前结束营业
    Then 系统应显示二次确认
    When 玩家取消
    Then 当前阶段仍为 SHOP_OPEN
    And 营业倒计时继续

  Scenario: 提前结算不重复加币
    Given 当前阶段为 SHOP_SETTLEMENT
    And shop_settlement_applied 为 true
    And shop_total_earned 为 120
    When 玩家关闭并重新打开存档
    Then 结算面板可以继续显示总收入 120
    And CurrencyWallet 不得再次增加 120 矿币

  Scenario: 未售出货物回仓
    Given 当前阶段为 SHOP_OPEN
    And 货台上有 sale_good_patch_kit x3
    When 玩家提前结束营业并确认
    Then sale_good_patch_kit x3 应回到仓库 AVAILABLE 状态
    And 出发准备和制造所后续可以重新读取该物资

  Scenario: 自然结束和提前结束共用结算管线
    Given 当前阶段为 SHOP_OPEN
    And 开店剩余时间为 0 秒
    When 系统进入营业结算
    Then 系统应生成 ShopSettlementSnapshot
    And shop_ended_by 应为 timer_end
    And 系统应复用提前结束的回仓和加币规则

  Scenario: 结算存在待回仓物资时不能进入夜晚
    Given 当前阶段为 SHOP_SETTLEMENT
    And pending_return_shelf_items 不为空
    When 玩家查看结算面板
    Then 进入夜晚按钮应置灰
    And 系统应提示仍有未回仓物资待处理

  Scenario: SHOP_OPEN 读档时剩余时间已归零
    Given 存档中的 outgame_phase 为 SHOP_OPEN
    And shop_elapsed_seconds 大于等于 shop_duration_seconds
    When 玩家读取存档
    Then 系统应直接进入 SHOP_SETTLEMENT 结算管线
    And 不应恢复到可操作的 SHOP_OPEN 界面

# 29_局外杂货店经营与界面重构路线图

> 文档版本：v0.1  
> 所属项目：《废土生存法则》  
> 关联文档：`09_局外仓库系统`、`10_研究所与制作所系统`、`11_出发准备与局外装备系统_修订版_废土生存法则.md`、`19_数值平衡表与掉落权重规范_修订版_废土生存法则.md`、`20_UI视觉规范与界面资源Sheet提示词_修订版_废土生存法则.md`、`23_世界观包装_废土生存法则.md`、`28_工程结构重构路线图.md`。  
> 文档目标：明确局外阶段的叙事包装、经济循环、杂货店订单系统、制造/转化/装备出口，以及 BaseScene UI 重构方向。

## 1. 一句话定位

玩家不是单纯的仓库管理员，也不是无身份的商人。

玩家是在末世避难所里经营一家小型杂货店的店主。避难所居民、工匠、巡逻队、医生和流动商队都会向店主提出需求。玩家白天或阶段间整理店铺、接订单、制造货品、分配库存；下一轮再冒险进入危险地表，带回能维持店铺和避难所运转的物资。

局外大循环：

```text
避难所订单/店铺需求
-> 出发准备
-> 局内搜集物资
-> 撤离结算入库
-> 杂货店上架/交付/制造/转化/研究/装备准备
-> 获得矿币、关系、设施成长和下一局能力
-> 新订单和更高风险目标
```

## 2. 设计参考与转译

参考方向：

- 类似“经营店铺”的资源加工和订单满足：玩家不是把物资直接变成钱，而是把物资变成避难所真正需要的货品。
- 类似“末世生存”的物资取舍：每个物品都有多种用途，卖掉、交付、制造、研究或留作装备都会互相竞争。

本项目不直接做餐厅或纯贸易市场，而是做：

```text
避难所杂货店
```

店铺不是轻松商业模拟，而是避难所物资调度中心。它提供更强的角色身份和更清晰的物资出口。

## 3. 玩家身份

玩家身份：

```text
避难所杂货店店主 / 回收人 / 临时物资调度员
```

玩家在避难所内拥有一个固定据点：

- 前台货架：展示可交付、可出售、可兑换的物资。
- 后仓：局外仓库。
- 工作台：制造所和资源转化。
- 账本：订单、居民需求、商队委托和经营记录。
- 装备墙：出发准备和装备选择。
- 图鉴柜：曾经收集过的道具记录。

叙事语气：

```text
不是“我要发财”，而是“这间店撑着一小群人活下去”。
```

## 4. 局外经济核心口径

局外资源出口分为五类。

| 出口 | 玩家行为 | 主要收益 | 设计作用 |
| --- | --- | --- | --- |
| 直接出售 | 把可出售物品卖给流动商队 | 少量矿币 | 兜底出口，避免废物资完全无用 |
| 居民订单 | 按居民需求交付指定物资或制成品 | 矿币、关系、剧情、解锁 | 类似店铺经营的核心 |
| 制造加工 | 消耗材料制作装备、消耗品、订单货品 | 下一局能力或高价值交付物 | 加工增值 |
| 同品级转化 | 用多个同品质资源换一个目标资源 | 补短板材料 | 缓解卡材料 |
| 研究/设施 | 消耗材料和矿币升级长期能力 | 永久成长 | 长线目标 |

原则：

- 直接出售永远收益最低。
- 订单交付收益高于直接出售。
- 制造加工后的交付收益最高，但需要玩家先投入材料和制造所能力。
- 不同出口互相竞争同一批仓库资源，形成取舍。

## 5. 货币与非货币收益

V0.2 保留 `mine_coin / 矿币`，但不让矿币成为唯一收益。

建议收益结构：

| 收益 | 用途 |
| --- | --- |
| 矿币 | 解锁制造所、购买稀缺物资、支付研究/设施升级费用 |
| 居民关系 | 解锁居民订单线、特殊图纸、剧情和折扣 |
| 店铺信誉 | 解锁更高等级订单、更多居民需求和商队访问 |
| 设施进度 | 解锁制造分类、货架容量、订单栏位、出发准备槽位 |
| 图纸/配方 | 解锁制造产物 |

V0.2 最小实现可以只做矿币和订单状态，关系/信誉先作为字段预留。

## 6. 订单系统

订单来源：

| 来源 | 需求风格 | 示例 |
| --- | --- | --- |
| 医生 | 医疗材料、绷带、药瓶、酒精棉 | “今晚前给诊所补 2 份简易绷带” |
| 巡逻队 | 装备、背包、照明、工具 | “需要 1 个小型加固背包” |
| 维修工 | 金属、电池、电路板、胶带 | “换气管线需要废金属 4、电池 1” |
| 居民家庭 | 食物、布料、生活材料 | “孩子发烧，需要药瓶和干净布料” |
| 流动商队 | 批量货品、稀有物资 | “收购滤芯 2 个，溢价支付矿币” |

订单字段建议：

```text
order_id
source_character_id
source_name
title
description
required_items
required_crafted_items
reward_currency_id
reward_currency_amount
reward_reputation
reward_relation
unlock_conditions
expires_after_runs
priority
repeat_policy
enabled_version
```

订单类型：

| 类型 | 规则 |
| --- | --- |
| 居民急需 | 限时 1-2 次出发，奖励不一定高，但影响关系或剧情 |
| 店铺常规需求 | 长期存在，用于引导基础材料出口 |
| 商队委托 | 高额矿币，需求量更大，偏中后期 |
| 制造交付 | 要求制成品，不接受原材料 |
| 剧情订单 | 推进章节或解锁设施 |

V0.2 最小实现：

```text
3 个固定居民订单 + 1 个商队订单栏位
订单不随机刷新或只在每次返回后简单刷新
订单交付即时完成
订单奖励只给矿币
```

## 7. 杂货店界面结构

BaseScene 顶部主导航建议改为：

```text
仓库
订单
制造所
研究所
商队
图鉴
出发准备
```

说明：

- “订单”是杂货店经营核心，放在仓库之后。
- “制造所”靠前，因为它和订单强关联。
- “商队”承接原商人功能，但从单纯卖货升级为出售、采购、委托。
- “出发准备”必须成为一级页签或常驻入口，装备系统上线后不能只靠开始按钮隐含处理。

V0.2 过渡期可以先保留现有顺序，但最终目标应迁移到以上结构。

常驻区域：

- 顶部：天数、矿币、店铺信誉/章节目标。
- 右上或底部：开始探索按钮。
- 中央：当前页签内容。
- 底部或侧边：当前章节目标、订单提醒、库存警告。

## 8. 制造所与资源转化

制造所解锁后开放两类能力。

### 8.1 配方制造

规则沿用 10 文档：

```text
仓库材料
-> 配方检查
-> 确认消耗
-> 产物入库
```

制造产物分为：

| 类型 | 用途 |
| --- | --- |
| 店铺货品 | 用于订单交付，通常不带入局内 |
| 消耗品 | 可在出发准备中携带 |
| 装备 | 可配置到出发准备 |
| 工具材料 | 用于研究/高级制造 |

### 8.2 同品级资源转化

目标：降低“只差一个材料”的挫败，给低价值物资一个稳定消耗口。

V0.2 建议规则：

```text
C 级：任意 2 个 C 级资源 -> 1 个指定 C 级基础资源
B 级：任意 3 个 B 级资源 -> 1 个指定 B 级基础资源
A 级：任意 4 个 A 级资源 -> 1 个指定 A 级基础资源
S/SS：不开放通用转化
```

示例：

```text
目标：废金属 C x1
消耗：脏布条 C x2
或：脏布条 C x1 + 小型加固背包 C x1
```

限制：

- 任务物品、图纸、货币、关键道具不可作为投入。
- SS 和收藏向道具不可作为普通投入。
- 装备作为投入时必须二次确认。
- 目标只能是基础资源或当前已在图鉴中见过的资源。
- 转化产物进入仓库，仓库空间不足时禁止转化。

## 9. 装备与出发准备

装备系统接入局外循环的关键不是“变强”，而是“持续消耗”。

最小闭环：

```text
制造或拾取装备
-> 仓库保存装备实例
-> 出发准备配置装备
-> 局内生效
-> 成功带回扣耐久
-> 耐久归零变 BROKEN
-> BROKEN 装备拆解成材料
-> 材料再进入制造或订单
```

V0.2 首版建议：

- 只做 BACK 背包装备。
- 装备提供负重或背包格子加成。
- 装备有耐久。
- 装备不能被研究/制造消耗，除非未配置到出发准备。
- BROKEN 装备不能出发，只能拆解。

## 10. 局外重构目标

当前 `scripts/base/base_scene.gd` 承担过多职责，后续局外系统扩展前必须拆分。

目标原则：

```text
BaseScene 只做局外场景壳层。
每个页签一个 Panel Controller。
每个业务系统一个 Manager/Service。
UI 不直接改仓库、货币、订单、装备和图鉴存档。
```

推荐拆分：

```text
scripts/base/base_scene.gd
scripts/base/base_tabs_controller.gd
scripts/base/base_warehouse_panel_controller.gd
scripts/base/base_order_panel_controller.gd
scripts/base/base_crafting_panel_controller.gd
scripts/base/base_research_panel_controller.gd
scripts/base/base_trade_panel_controller.gd
scripts/base/base_catalog_panel_controller.gd
scripts/base/base_loadout_panel_controller.gd
scripts/base/base_debug_panel_controller.gd
scripts/ui/item_tooltip_service.gd
```

业务层推荐：

```text
scripts/game/order_service.gd
scripts/game/trade_service.gd
scripts/game/crafting_manager.gd
scripts/game/item_conversion_service.gd
scripts/game/equipment_manager.gd
scripts/game/loadout_manager.gd
```

拆分顺序：

1. 为现有仓库、商人、研究、制造解锁、图鉴建立回归测试或最小验证。
2. 抽 BaseTabsController，只负责切页、锁定、页签高亮。
3. 抽 TooltipService，避免每个面板复制 hover 逻辑。
4. 抽 WarehousePanel、TradePanel、ResearchPanel、CatalogPanel。
5. 再新增 OrderPanel、CraftingPanel、LoadoutPanel。

## 12. V0.2 最小交付建议

第一批不要试图一次做完整经营模拟。

建议 V0.2 最小交付：

```text
局外 UI 拆分出页签控制器和独立面板
制造所解锁后可进入制造页
实现 3-5 个固定配方
实现 C/B/A 同品级资源转化
实现订单页，至少 3 个固定订单
订单交付消耗仓库物品并奖励矿币
实现出发准备页
实现 BACK 装备槽和 1-2 个背包装备
装备成功带回扣耐久
BROKEN 装备可拆解
```

暂不实现：

- 复杂顾客排队。
- 实时店铺营业。
- 多 NPC 动画和站位。
- 复杂价格波动。
- 声望多分支。
- 随机词条装备。
- 多设备制造队列。

## 13. 体验目标

玩家每次回到局外时，应该自然产生三个问题：

```text
谁需要我这次带回来的东西？
我应该把它卖掉、交订单、做成装备，还是留着研究？
下一局我为了哪个订单或制造目标出发？
```

如果这三个问题成立，局外循环就不是菜单堆叠，而是一个可持续的经营循环。

## BDD 场景补充

```gherkin
Feature: 避难所杂货店局外循环
  作为 避难所杂货店店主
  我希望把地表带回的物资用于订单、制造、交易、研究和出发准备
  以便每次局内探索都服务于明确的局外经营目标

  Background:
    Given 玩家已经创建本地档案
    And 玩家可以进入局外 BaseScene
    And 局外仓库、矿币和道具表已加载

  Scenario: 居民订单提供明确搜集目标
    Given 订单页存在居民订单 order_clinic_bandage
    And 订单要求简易绷带 x2
    When 玩家查看订单页
    Then 界面应显示订单来源、需求物品、奖励和交付状态
    And 玩家应能把该订单作为下一局搜集或制造目标

  Scenario: 订单交付消耗仓库物品并奖励矿币
    Given 仓库中有 order_clinic_bandage 要求的物品
    When 玩家在订单页点击交付并确认
    Then 系统应从仓库扣除对应物品
    And 系统应增加订单定义的矿币奖励
    And 订单状态变为已完成

  Scenario: 材料不足时不能交付订单
    Given 仓库中缺少 order_clinic_bandage 要求的物品
    When 玩家查看该订单
    Then 交付按钮应置灰或显示材料不足
    And 点击交付不得扣除仓库物品或增加矿币

  Scenario: 同品级资源转化解决短缺材料
    Given 玩家需要废金属 C x1
    And 仓库中有脏布条 C x2
    When 玩家在制造所选择 C 级资源转化为废金属
    Then 系统应消耗脏布条 C x2
    And 系统应向仓库加入废金属 C x1

  Scenario: 装备形成制造和材料消耗闭环
    Given 玩家制造了小型加固背包
    And 玩家在出发准备中装备该背包
    When 玩家成功撤离并带回该装备
    Then 该装备应扣除耐久
    And 耐久为 0 时装备变为 BROKEN
    And BROKEN 装备可拆解为材料
```

验收方式：

- 文本验证：检查本文是否保留 `Feature`、`Scenario`、`Given`、`When`、`Then`。
- 文本验证：使用 `rg -n "BaseTabsController|OrderPanel|order_service|item_conversion|LoadoutPanel" scripts scenes Doc` 检查实现路径。
- Debug 验证：通过 Debug 面板添加订单材料、完成订单、执行资源转化、制造装备、模拟装备扣耐久。
- 手动验收：新档进入局外后，确认玩家能理解“杂货店店主”身份，并能从订单、制造、出发准备形成下一局目标。

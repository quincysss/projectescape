# 00_拆分规范与落地文档写法

> 来源文档：`../02_核心循环规则_修订版_废土生存法则.md`  
> 目标：定义 02 系统后续拆分文档的样式。03-14 的拆分工作如果没有特殊原因，应复用本规范。

---

## 0. 执行总则

本目录所有拆分文档必须遵守：

```text
Doc/00_落地执行文档总则.md
```

如果本文档与总则冲突，以总则为准；如果具体模块文档与本文档冲突，以更具体、可验证、已落地的规则为准。

---

## 1. 拆分原则

原文档保留为设计母版，不删除、不改名。拆分文档不是复制原文条目，而是把原文中的规则转成 Godot 可执行任务。

每份拆分文档必须满足：

| 要求 | 说明 |
|---|---|
| 单一职责 | 一份文档只承载一个实现切片，例如状态机、初始化、容器交互、撤离结算 |
| 可开工 | 读完后可以直接创建场景、脚本、资源或测试，不需要再反推实现边界 |
| 规则足够细 | 必须把模块内的触发、限制、边界、失败、反馈、调试规则写清楚 |
| 可验收 | 必须写清楚完成标准、最小测试场景、调试入口 |
| 可串联 | 明确依赖哪些文档、产出会被哪些后续文档使用 |
| 不抢专业文档 | 02 只定义核心循环编排，具体数值、掉落权重、局外仓库、音乐等细节引用对应系统文档 |

---

## 2. 推荐文档结构

每份落地文档使用以下结构：

```text
# 文档名

## 1. 实现目标
说明这个切片要让玩家/系统获得什么能力。

## 2. 设计依据
列出来自母版文档的规则，不逐字复制，只保留实现必须遵守的约束。

## 3. 功能规则细化
把模块内部规则拆到可实现粒度：
- 触发条件
- 允许行为
- 禁止行为
- 边界情况
- 失败/中断处理
- UI/反馈要求
- 调试要求

## 4. Godot 落地范围
列出建议新增或修改的 scenes、scripts、resources、autoload。

## 5. 数据结构
列出枚举、字段、配置资源、运行时状态。

## 6. 核心流程
用步骤、状态转换或伪代码描述实际运行逻辑。

## 7. 信号与接口
列出要发出的 signal、要监听的事件、跨系统调用边界。

## 8. AI/MCP 开工提示词
给未来执行代理一段可以直接复制使用的任务提示词。提示词必须包含：
- 任务目标
- 读取哪些文档
- 允许新增/修改哪些文件
- 禁止实现哪些范围
- 完成标准
- 验证方式
- 汇报格式

## 9. 验收标准
列出人工或自动测试可以验证的结果。

## 10. 暂不实现
明确 V0.1 不做的内容，避免实现发散。
```

## 4. Godot 约定

在实际项目结构确定前，02 文档统一使用建议路径：

| 类型 | 建议路径 |
|---|---|
| 局内主场景 | `res://scenes/run/RunScene.tscn` |
| 玩家 | `res://scenes/player/Player.tscn` |
| 核心循环脚本 | `res://scripts/run/RunDirector.gd` |
| 状态机 | `res://scripts/run/RunStateMachine.gd` |
| 安全区 | `res://scripts/run/SafeZone.gd` |
| 稳定值 | `res://scripts/player/StabilityComponent.gd` |
| 视野遮罩 | `res://scripts/vision/VisionController.gd` |
| 容器 | `res://scripts/loot/LootContainer.gd` |
| 前哨材料 | `res://scripts/outpost/OutpostMaterialPickup.gd` |
| 前哨站 | `res://scripts/outpost/OutpostStation.gd` |
| 背包负重 | `res://scripts/inventory/InventoryComponent.gd` |
| 撤离结算 | `res://scripts/run/ExtractionController.gd` |
| 调试场景 | `res://scenes/debug/DebugCoreLoop.tscn` |

如果已有项目结构不同，执行时优先贴合现有结构，但保持职责边界不变。

### 4.1 V0.1 启动入口与调试入口

所有 02 核心循环落地任务默认遵守以下入口规则：

| 用途 | 场景 | 说明 |
|---|---|---|
| 项目正式初始场景 | `res://scenes/boot/BootScene.tscn` | 必须配置到 `project.godot` 的 `application/run/main_scene` |
| 局外基地入口 | `res://scenes/base/BaseScene.tscn` | V0.1 可由 BootScene 直接进入 |
| 局内正式场景 | `res://scenes/run/RunScene.tscn` | BaseScene 点击开始探索后进入 |
| 核心循环调试场景 | `res://scenes/debug/DebugCoreLoop.tscn` | 只用于验证状态机、初始化、成功/失败路径 |

落地线程开工前必须先检查：

```text
[ ] BootScene.tscn 存在。
[ ] BaseScene.tscn 存在。
[ ] RunScene.tscn 存在。
[ ] DebugCoreLoop.tscn 存在。
[ ] project.godot 已设置 run/main_scene="res://scenes/boot/BootScene.tscn"。
```

如果 `run/main_scene` 缺失，优先补上启动配置，再继续做玩法模块。

禁止事项：

```text
不要把 DebugCoreLoop 当正式初始场景。
不要让模块脚本依赖“当前运行的是哪个调试场景”。
不要在 BootScene 中堆放稳定值、背包、容器、前哨等业务逻辑。
```

## 5. 状态与事件命名

核心循环状态统一使用大写枚举：

```gdscript
enum RunPhase {
    SPAWN,
    OBSERVE,
    LEAVE_HOME,
    SCAVENGE,
    RECOVER,
    OUTPOST_PUSH,
    GREED_DECISION,
    EXTRACT,
    SETTLEMENT,
    FAILED
}
```

跨系统事件优先使用信号，而不是让系统互相硬引用：

```text
phase_changed(old_phase, new_phase)
safe_zone_entered(zone_id)
safe_zone_exited(zone_id)
stability_changed(current, max_value, stage)
stability_depleted()
container_open_started(container_id)
container_opened(container_id, rewards)
material_picked(material_id, amount)
weight_changed(current_weight, max_weight, stage)
outpost_repaired(outpost_id)
extraction_unlocked()
run_finished(result)
```

## 6. MCP 开工指令写法

每份文档中的 MCP 指令应像这样写：

```text
你是 Godot 项目的执行代理。请实现 Doc/02_核心循环规则/01_单局流程状态机.md 中定义的“单局流程状态机”。

必须读取：
- Doc/02_核心循环规则/00_拆分规范与落地文档写法.md
- Doc/02_核心循环规则/01_单局流程状态机.md
- Doc/02_核心循环规则/11_V0_1最小可玩闭环任务清单.md

实现范围：
- 只实现状态机、RunDirector 的最小编排、调试入口。
- 可以新增 scripts/run 与 scenes/debug 下的必要文件。

禁止范围：
- 不做正式美术。
- 不做正式 UI。
- 不做多人同步。
- 不做局外仓库。

完成标准：
- 调试入口能模拟从 SPAWN 到 SETTLEMENT 的成功流程。
- 稳定值归零事件能进入 FAILED。
- 非法状态切换能输出原因。

验证方式：
- 运行 Godot 可用检查命令或打开调试场景验证。
- 汇报新增文件、主要信号、验证结果、未实现项。
```

指令要避免“做得更丰富”“优化体验”这类模糊目标，改成可验证动作。

## 7. 功能规则细化写法

“功能规则细化”必须比母版更接近实现。推荐使用以下表格：

| 分类 | 写法 |
|---|---|
| 触发条件 | 什么事件发生后开始执行 |
| 前置条件 | 执行前必须满足哪些状态 |
| 执行规则 | 执行中每一步如何变化 |
| 中断规则 | 玩家离开、取消、死亡、状态改变时如何处理 |
| 失败规则 | 条件不满足时如何反馈，不允许静默失败 |
| 反馈规则 | UI、音效、日志、调试可视化至少要有一种 |
| 数据规则 | 哪些字段被读取、写入、清空 |
| 边界规则 | 重复触发、空数据、数值越界、候选点不足如何处理 |

示例：

```text
容器开启规则不能只写“玩家按 F 开箱”。
必须写清：
- 只有 AVAILABLE 状态可开。
- 玩家必须在交互范围内。
- 开启中离开范围则取消。
- 开启中容器生命周期不删除。
- 完成后只生成一次奖励。
- 背包或负重不足时奖励仍留在容器结果中。
```

## 8. AI 提示词质量底线

每份 AI/MCP 提示词必须满足：

```text
能单独复制给执行代理使用
不依赖口头上下文
明确要读哪些文件
明确允许改哪些范围
明确禁止做哪些范围
明确完成后怎么验证
明确最终报告格式
```

禁止提示词写成：

```text
帮我完善这个系统
做得细一点
优化一下体验
按文档实现
```

合格提示词应该写成：

```text
请实现“家安全区观察与离家切换”。
读取 03_家安全区观察与离家切换.md。
新增 SafeZone/HomeSafeZone/RunCameraController 的最小实现。
玩家进入家时必须停止稳定值衰减、关闭遮罩、切到观察镜头；离家时必须开启遮罩、切到跟随镜头、恢复稳定值衰减。
禁止实现正式 UI 和美术。
完成后用 DebugCoreLoop 场景验证进入/离开家各 3 次，确认状态不会重复错乱。
```

## 9. 质量底线

落地文档必须避免：

```text
只重复原文规则
只写概念不写字段
只写玩法不写 Godot 节点
只写最终效果不写状态转换
没有验收标准
没有 AI/MCP 可复制提示词
把多个大系统塞进一份文档
```

合格的文档读起来应该像“开发任务说明 + 设计约束 + 验收单”，而不是新的设定稿。

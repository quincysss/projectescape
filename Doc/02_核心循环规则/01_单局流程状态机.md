# 01_单局流程状态机

> 对应母版章节：2、3、18、19、20、21  
> 实现层级：核心编排层。所有局内系统都应向状态机汇报事件，由状态机决定阶段切换。
> 施工补充：开工前必须读取 `12_模块功能规则细化与AI开工提示词.md` 的“单局流程状态机”小节。

---

## 1. 实现目标

建立 V0.1 单人局内核心闭环的总状态机，让玩家可以从“出生在家”推进到“撤离结算”或“死亡失败”。状态机不负责具体玩法细节，只负责阶段、事件、准入条件和跨系统编排。

---

## 2. 设计依据

V0.1 单局由 9 个阶段构成：

| 阶段 | 状态名 | 进入条件 | 离开条件 |
|---|---|---|---|
| 出生 | `SPAWN` | 创建新局 | 初始化完成 |
| 观察 | `OBSERVE` | 玩家位于家安全区 | 玩家离开家安全区 |
| 离家 | `LEAVE_HOME` | 首次离开安全区 | 镜头与遮罩切换完成 |
| 搜集 | `SCAVENGE` | 玩家处于危险区 | 进入安全区、死亡、满足撤离目标 |
| 回收 | `RECOVER` | 进入家或已修复前哨站 | 玩家再次离开安全区 |
| 前哨推进 | `OUTPOST_PUSH` | 材料满足并开始修复 | 修复完成或取消 |
| 贪婪决策 | `GREED_DECISION` | 两个前哨站均修复 | 玩家继续搜集或返回家撤离 |
| 撤离 | `EXTRACT` | 撤离解锁并在家触发 | 读条完成或中断 |
| 结算 | `SETTLEMENT` | 撤离成功或失败进入结果处理 | 返回局外 |

死亡可从危险区、开箱读条、超时探索等阶段进入 `FAILED`。

---

## 3. Godot 落地范围

建议新增：

| 路径 | 职责 |
|---|---|
| `res://scripts/run/RunStateMachine.gd` | 阶段枚举、切换条件、切换信号 |
| `res://scripts/run/RunDirector.gd` | 持有局内系统引用，转发事件给状态机 |
| `res://scenes/run/RunScene.tscn` | 局内主场景，挂载 `RunDirector` |
| `res://scenes/debug/DebugCoreLoop.tscn` | 用按钮或热键模拟核心事件 |

`RunStateMachine` 不应直接操作玩家、UI、容器、前哨站节点。它只发信号，由 `RunDirector` 或具体系统响应。

## 4. 数据结构

```gdscript
enum RunPhase {
    SPAWN,
    OBSERVE,
    LEAVE_HOME,
    SCAVENGE,
    RECOVER,
    OUTPOST_PUSH,
    GREED_DECISION,
    EXTRACT,
    SETTLEMENT,
    FAILED
}

enum RunResult {
    NONE,
    EXTRACTED,
    DEAD,
    ABORTED
}
```

运行时字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `current_phase` | `RunPhase` | 当前阶段 |
| `previous_phase` | `RunPhase` | 上一个阶段 |
| `run_result` | `RunResult` | 本局结果 |
| `is_extraction_unlocked` | `bool` | 两个前哨站修复后置真 |
| `repaired_outpost_count` | `int` | 已修复前哨站数量 |
| `is_player_in_safe_zone` | `bool` | 家或已修复前哨站均算安全区 |
| `is_player_alive` | `bool` | 死亡后不可再切回局内阶段 |

## 5. 核心流程

启动流程：

```text
RunDirector.start_new_run()
→ 初始化局内上下文
→ 状态机进入 SPAWN
→ 初始化完成后进入 OBSERVE
→ 玩家离开家安全区后进入 LEAVE_HOME
→ 镜头和黑暗遮罩完成后进入 SCAVENGE
```

搜集与安全区流程：

```text
SCAVENGE
→ 玩家进入家或已修复前哨站：RECOVER
→ 玩家离开安全区：SCAVENGE
→ 玩家开始修复前哨站：OUTPOST_PUSH
→ 修复完成：RECOVER 或 SCAVENGE，取决于玩家是否在前哨安全区内
```

目标完成流程：

```text
outpost_repaired_count == 2
→ is_extraction_unlocked = true
→ 状态机允许 GREED_DECISION
→ 玩家回家并按撤离按钮
→ EXTRACT
→ 读条完成
→ SETTLEMENT
```

失败流程：

```text
stability_depleted
→ FAILED
→ run_result = DEAD
→ 进入失败结算或死亡处理
```

## 6. 信号与接口

`RunStateMachine` 发出：

```gdscript
signal phase_changed(old_phase: int, new_phase: int)
signal extraction_unlocked()
signal run_finished(result: int)
signal invalid_transition_requested(from_phase: int, to_phase: int, reason: String)
```

`RunDirector` 监听或接收：

```text
on_run_initialized()
on_home_exited()
on_safe_zone_entered(zone_id)
on_safe_zone_exited(zone_id)
on_camera_transition_finished()
on_outpost_repair_started(outpost_id)
on_outpost_repaired(outpost_id)
on_extraction_started()
on_extraction_completed()
on_player_dead(reason)
```

## 8. 验收标准

| 编号 | 标准 |
|---|---|
| A1 | 新局启动后必定先进入 `SPAWN`，初始化完成后进入 `OBSERVE` |
| A2 | 离开家会触发 `LEAVE_HOME`，切换完成后进入 `SCAVENGE` |
| A3 | 进入安全区时进入 `RECOVER`，离开安全区后回到 `SCAVENGE` |
| A4 | 修复两个前哨站后发出 `extraction_unlocked` |
| A5 | 撤离读条完成后进入 `SETTLEMENT`，结果为 `EXTRACTED` |
| A6 | 稳定值归零后进入 `FAILED`，结果为 `DEAD` |
| A7 | 非法阶段切换不会静默发生，必须输出原因或发出信号 |

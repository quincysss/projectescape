# 12_模块功能规则细化与AI开工提示词

> 来源文档：`../02_核心循环规则_修订版_废土生存法则.md`  
> 用途：补足 02 拆分文档中的“施工级细节”和“可直接喂给 AI/MCP 的提示词”。  
> 使用方式：未来执行某个模块时，先读对应模块文档，再读本文件中同名模块的小节。

---

## 1. 使用规则

本文件不是新的设计母版，而是执行增强层。它解决两个问题：

```text
1. 模块规则必须细到可以直接写代码。
2. AI/MCP 提示词必须能独立复制使用，不依赖口头补充。
```

每个模块都按以下格式补强：

| 栏目 | 作用 |
|---|---|
| 功能规则细化 | 触发、前置、执行、中断、失败、反馈、边界 |
| AI/MCP 开工提示词 | 可直接复制给执行代理 |
| 完成报告要求 | 执行代理最后必须汇报什么 |

## 2. 单局流程状态机

对应文档：`01_单局流程状态机.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | `RunDirector.start_new_run()` 被调用后创建状态机并进入 `SPAWN` |
| 前置条件 | 不能已有进行中的未结束局；如已有，需要先明确结束旧局 |
| 执行规则 | 所有阶段切换必须通过 `request_transition()` 或等价入口，不允许外部直接改 `current_phase` |
| 中断规则 | 撤离读条中断回到 `RECOVER` 或 `OBSERVE`，死亡事件优先级高于普通阶段切换 |
| 失败规则 | 非法切换不改变状态，必须输出 `from_phase`、`to_phase`、`reason` |
| 反馈规则 | 每次阶段变化至少打印日志；调试 HUD 可显示当前阶段 |
| 数据规则 | 记录 `previous_phase`、`current_phase`、`run_result`、`repaired_outpost_count` |
| 边界规则 | `FAILED`、`SETTLEMENT` 进入后不可回到探索阶段，除非启动新局 |

状态机必须只做编排，不直接移动玩家、不直接改 UI、不直接生成物品。

### AI/MCP 开工提示词

```text
你是 Godot 项目的执行代理。请实现“单局流程状态机”。

必须读取：
- Doc/02_核心循环规则/00_拆分规范与落地文档写法.md
- Doc/02_核心循环规则/01_单局流程状态机.md
- Doc/02_核心循环规则/12_模块功能规则细化与AI开工提示词.md 的“单局流程状态机”小节

实现范围：
- 新增或修改 RunStateMachine.gd、RunDirector.gd、DebugCoreLoop.tscn 相关最小文件。
- 实现 RunPhase、RunResult、阶段切换入口、非法切换反馈、phase_changed/run_finished/extraction_unlocked 等信号。
- 提供调试入口模拟：初始化、离家、进入安全区、修复前哨、撤离、死亡。

禁止范围：
- 不实现正式 UI。
- 不实现容器、背包、前哨站的具体玩法。
- 不实现多人同步。

完成标准：
- 调试入口能走通 SPAWN -> OBSERVE -> LEAVE_HOME -> SCAVENGE -> RECOVER -> EXTRACT -> SETTLEMENT。
- 稳定值归零模拟事件能进入 FAILED。
- 非法切换不会改变阶段，并能输出原因。

验证方式：
- 运行可用的 Godot 检查命令，或打开 DebugCoreLoop.tscn 手动验证。
- 报告新增/修改文件、信号列表、验证结果、未实现项。
```

## 3. 单局开始与初始化

对应文档：`02_单局开始与初始化.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 玩家从局外入口或调试按钮调用 `start_new_run()` |
| 前置条件 | 场景中必须存在玩家、家出生点、前哨候选点、基础配置 |
| 执行规则 | 初始化必须一次性填充 `RunContext`，再通知其他系统读取 |
| 中断规则 | 候选点不足、出生点缺失、配置缺失时初始化失败，不进入 `OBSERVE` |
| 失败规则 | 失败必须返回明确原因，不允许半初始化继续运行 |
| 反馈规则 | 初始化成功后打印 run_id、seed、出生点、前哨点、撤离状态 |
| 数据规则 | `RunContext` 是单局事实来源，不把随机结果散落在多个系统 |
| 边界规则 | 重开新局必须清理上一局临时数据、容器、材料、前哨状态 |

### AI/MCP 开工提示词

```text
请实现“单局开始与初始化”。

必须读取：
- Doc/02_核心循环规则/02_单局开始与初始化.md
- Doc/02_核心循环规则/12_模块功能规则细化与AI开工提示词.md 的“单局开始与初始化”小节

实现范围：
- 新增 RunContext、RunConfig、RunInitializer。
- RunDirector.start_new_run() 必须能创建完整 RunContext。
- 初始化玩家出生点、稳定值、背包、负重、家存储、前哨候选点、容器刷新状态、撤离状态、镜头/遮罩初始状态。

禁止范围：
- 不接正式局外装备配置。
- 不做正式地图生成。
- 不写永久存档。

完成标准：
- 新局启动后玩家位于家出生点。
- 稳定值 100，黑暗遮罩关闭，撤离未解锁。
- 第一前哨从 3 个候选点选 1 个，第二前哨从 4 个候选点选 1 个。
- 候选点不足时初始化失败并报告原因。

验证方式：
- 用调试场景连续新建 3 局，确认每局 RunContext 完整且旧数据清理。
- 汇报新增文件、初始化日志示例、失败分支验证结果。
```

## 4. 家安全区观察与离家切换

对应文档：`03_家安全区观察与离家切换.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 玩家进入或离开 `HomeSafeZone` 的检测区域 |
| 前置条件 | 玩家节点可被安全区识别，安全区处于激活状态 |
| 执行规则 | 在家：停稳定值衰减、恢复稳定值、关闭遮罩、切观察镜头、允许看信号 |
| 执行规则 | 离家：切玩家跟随镜头、开启遮罩、开始稳定值衰减、关闭全图观察 |
| 中断规则 | 镜头切换中玩家反复进出安全区时，以最后一次区域状态为准 |
| 失败规则 | 摄像机或视野控制器缺失时仍切换状态，但必须输出警告 |
| 反馈规则 | 调试 HUD 显示 `in_home_safe_zone`、camera mode、darkness enabled |
| 边界规则 | 玩家出生时如果已经在家中，不应重复触发离家逻辑 |

### AI/MCP 开工提示词

```text
请实现“家安全区观察与离家切换”。

- 新增 SafeZone/HomeSafeZone 的最小实现。
- 接入 RunDirector，使玩家进入家时进入 OBSERVE 或 RECOVER，离家时进入 LEAVE_HOME/SCAVENGE。
- 接入或预留 CameraController、VisionController、StabilityComponent 的调用入口。

禁止范围：
- 不实现正式信号图标。
- 不实现完整家中整理 UI。
- 不实现正式镜头美术表现。

完成标准：
- 玩家在家时稳定值不下降、遮罩关闭、镜头为观察模式。
- 玩家离家时遮罩开启、镜头跟随玩家、稳定值开始下降。
- 玩家快速进出家安全区不会造成状态错乱。

验证方式：
- 在调试场景进出家安全区 3 次，记录状态机、镜头、遮罩、稳定值状态。
- 汇报新增文件、信号、验证过程。
```

## 5. 稳定值与视野闭环

对应文档：`04_稳定值与视野闭环.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 离开安全区开始衰减，进入安全区开始恢复 |
| 前置条件 | 稳定值组件已初始化最大值和当前值 |
| 执行规则 | 危险区每秒下降，安全区每秒恢复，数值 clamp 到 0-100 |
| 阶段规则 | 跨过 75、50、25、0 阈值时更新阶段并发信号 |
| 中断规则 | 死亡后停止继续衰减和重复死亡信号 |
| 失败规则 | 视野控制器缺失时稳定值仍运行，并输出警告 |
| 反馈规则 | HUD 显示数值和阶段；视野半径随阶段变化 |
| 边界规则 | delta 异常或低帧率不能导致数值跳过死亡处理 |

```text
请实现“稳定值与视野闭环”。

- 稳定值归零只触发一次死亡事件。

验证方式：
- 用调试按键降低/恢复稳定值，验证阶段阈值和死亡。
- 汇报新增文件、阈值、验证日志。
```

## 6. 容器搜集交互闭环

对应文档：`05_容器搜集交互闭环.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 玩家进入容器交互范围，按交互键开始开启 |
| 前置条件 | 容器状态必须是 `AVAILABLE`，玩家处于可搜集阶段 |
| 执行规则 | 开启进入 `OPENING`，显示读条，完成后进入 `OPENED` 并生成一次奖励 |
| 中断规则 | 玩家离开范围、死亡、阶段切走时取消读条 |
| 锁定规则 | 容器 `OPENING` 时生命周期不删除，取消后如已过期再删除 |
| 失败规则 | 背包格子或负重不足时不可拾取，但奖励不消失 |
| 反馈规则 | 靠近提示、读条、开启完成、拾取失败原因 |
| 边界规则 | 已开启容器不能二次生成奖励；同一玩家不能同时开两个容器 |

## 9. 资源刷新与异常预兆编排

对应文档：`08_资源刷新与异常预兆编排.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 新局初始化完成后启动刷新调度 |
| 前置条件 | 点位池存在且可识别占用状态 |
| 执行规则 | 容器按刷新间隔和上限生成；材料按当前需求生成，120 秒未拾取则换点重刷 |
| 中断规则 | 本局结束时清理所有局内刷新物 |
| 失败规则 | 无可用点位时输出警告，不崩溃 |
| 反馈规则 | 调试日志显示刷新点位、资源 ID、移除原因 |
| 数据规则 | 点位占用必须在资源生成/移除时同步更新 |
| 边界规则 | 正在开启的容器不能被过期删除 |

### AI/MCP 开工提示词

```text
请实现“资源刷新与异常预兆编排”的最小版本。

必须读取：
- Doc/02_核心循环规则/08_资源刷新与异常预兆编排.md
- Doc/02_核心循环规则/12_模块功能规则细化与AI开工提示词.md 的“资源刷新与异常预兆编排”小节

实现范围：
- 新增 ResourceRefreshDirector、LootSpawnPoint、MaterialSpawnPoint。
- 容器按固定间隔生成到空闲点位，过期后释放点位。
- 当前前哨材料按需求生成，120 秒未拾取则移除并立刻换点重刷。
- S 级异常只做预留信号和调试日志。

禁止范围：
- 不实现正式刷新权重。
- 不实现正式掉落表。
- 不实现异常奖励和危险事件。

完成标准：
- 容器只生成在空闲点位。
- 未开启容器过期后释放点位。
- 正在开启的容器不被过期删除。
- 材料生成后 120 秒内可拾取；过期后换到另一个合法点位重刷。
```

## 11. 撤离死亡结算闭环

对应文档：`10_撤离死亡结算闭环.md`

### 功能规则细化

| 分类 | 规则 |
|---|---|
| 触发条件 | 两个前哨修复后解锁撤离；稳定值归零触发死亡 |
| 前置条件 | 撤离必须在家中触发，死亡可从危险区直接触发 |
| 执行规则 | 撤离读条完成进入结算；死亡进入失败结算 |
| 中断规则 | 撤离读条中离开家、死亡或取消会中断 |
| 失败规则 | 未解锁或不在家中时按撤离无效处理，不能开始读条 |
| 反馈规则 | 撤离按钮状态、读条、中断原因、结算列表 |
| 数据规则 | 撤离成功带回背包和家中存储；死亡丢失背包、保留家中存储 |
| 边界规则 | 本局结束后禁止再次触发撤离、死亡、开箱、修复 |

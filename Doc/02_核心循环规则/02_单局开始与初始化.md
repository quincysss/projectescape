# 02_单局开始与初始化

> 对应母版章节：4、13.1、18.1、20  
> 实现层级：新局上下文创建。
> 施工补充：开工前必须读取 `12_模块功能规则细化与AI开工提示词.md` 的“单局开始与初始化”小节。

---

## 1. 实现目标

玩家点击开始探索后，系统创建一局可运行的局内上下文：玩家出生在家，稳定值、背包、负重、容器刷新、前哨站候选点、撤离状态和本局计时器全部进入确定初始状态。

新增场景随机事件后，单局初始化还需要读取 `RunEventContext`，决定本局是否生成怪物、是否改写对局时长、是否生成随机障碍。完整规则见 `Doc/26_怪物与场景随机事件系统`。

---

## 2. 设计依据

单局初始化必须创建这些内容：

```text
玩家位置
玩家稳定值
玩家背包
玩家当前负重
玩家负重上限
家中临时存储
前哨站候选点抽取结果
前哨站状态
前哨材料需求
前哨材料点位池
容器刷新计时器
容器点位池
场景随机事件结果
怪物事件是否激活
怪物街道刷新点抽取结果
时间事件是否激活
本局实际对局时长
随机障碍事件是否激活
随机障碍点位抽取结果
撤离状态
镜头状态
黑暗遮罩状态
本局计时器
多人预留字段
```

V0.1 如果局外配置未完成，玩家默认空背包、基础负重、稳定值 100。

## 4. 数据结构

`RunConfig` 建议字段：

| 字段 | 默认值 | 说明 |
|---|---:|---|
| `max_stability` | `100` | 稳定值上限 |
| `base_weight_limit` | `20` | 基础负重上限，最终数值以后接 07 文档 |
| `home_storage_slots` | `1` | 家中临时存储格；研究后提高至 2 / 3 / 4 |
| `first_outpost_storage_slots` | `0` | 前哨临时存储格；未研究时不激活前哨存储 |
| `second_outpost_storage_slots` | `0` | 前哨临时存储格；未研究时不激活前哨存储 |
| `first_outpost_candidate_count` | `3` | 第一前哨站候选池数量 |
| `second_outpost_candidate_count` | `4` | 第二前哨站候选池数量 |
| `container_refresh_interval` | 待 08 文档定 | 容器刷新节奏 |
| `run_seed` | 随机 | 本局随机种子 |
| `run_day_index` | `surface_day + 1` | 即将进入的地表日 |
| `base_run_duration_seconds` | `300` | 默认对局时长，5 分钟 |
| `run_duration_seconds` | `300` | 本局实际对局时长，可能被时间事件改写 |
| `monster_spawn_point_count` | `10` | 怪物点位总数 |
| `monster_active_count` | `4` | 怪物事件命中时生成数量 |
| `obstacle_spawn_point_count` | `10` | 随机障碍候选点总数 |
| `obstacle_count_min` | `2` | 障碍事件命中时最少生成数量 |
| `obstacle_count_max` | `4` | 障碍事件命中时最多生成数量 |

`RunContext` 建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `run_id` | `String` | 本局唯一 ID |
| `seed` | `int` | 本局随机种子 |
| `elapsed_seconds` | `float` | 本局时长 |
| `run_duration_seconds` | `float` | 本局倒计时总时长，默认 300，可由事件改写 |
| `player_spawn_position` | `Vector2/Vector3` | 由项目维度决定 |
| `player_stability` | `float` | 当前稳定值 |
| `player_inventory` | `Array` | 背包物品 |
| `home_storage` | `Array` | 家中临时存储 |
| `selected_outposts` | `Array` | 本局前哨站节点或 ID |
| `outpost_states` | `Dictionary` | 前哨站状态 |
| `scene_events` | `Dictionary` | 本局场景随机事件结果 |
| `active_time_event_id` | `StringName` | 本局时间事件，空值表示默认 5 分钟 |
| `monster_event_active` | `bool` | 本局是否生成怪物 |
| `monster_spawn_point_ids` | `Array` | 本局抽中的怪物点位 |
| `obstacle_event_active` | `bool` | 本局是否生成随机障碍 |
| `obstacle_spawn_point_ids` | `Array` | 本局抽中的障碍候选点 |
| `is_extraction_unlocked` | `bool` | 撤离是否解锁 |

## 5. 核心流程

```text
1. RunDirector 收到 start_new_run。
2. RunInitializer 读取 RunConfig。
3. 生成 run_id 与 seed。
4. 寻找 HomeSpawnPoint，设置玩家位置与朝向。
5. 初始化玩家稳定值为 100。
6. 初始化背包为空或读取局外出发配置。
7. 计算当前负重与负重上限。
8. 初始化家中临时存储为 4 格。
9. 从第一前哨候选池随机 1 个，标记为未修复。
10. 从第二前哨候选池随机 1 个，标记为未修复。
11. 根据当前前哨需求生成材料目标。
12. 初始化容器刷新计时器和点位池。
13. SceneRandomEventDirector 解析本局场景随机事件。
14. 根据时间事件写入 run_duration_seconds：默认 300，超级时间 480，超短时间 210。
15. 如果怪物事件激活，从 10 个怪物街道点位中抽 4 个。
16. 如果随机障碍事件激活，从 10 个障碍点中抽 2-4 个并做路径校验。
17. RunTimerController 使用 run_duration_seconds 初始化倒计时。
18. 撤离按钮置为未激活。
19. 镜头进入全图观察模式，黑暗遮罩关闭。
20. 状态机进入 SPAWN，然后进入 OBSERVE。
```

随机规则：

```text
同一局内的候选点抽取结果固定。
重开新局可以重新随机。
如果候选点数量不足，必须输出错误并阻止进入正式局内。
```

## 6. 信号与接口

初始化完成后发出：

```gdscript
signal run_context_created(context)
signal run_initialized(context)
signal initialization_failed(reason: String)
```

需要通知：

```text
RunStateMachine：进入 SPAWN/OBSERVE
Player：设置位置、背包、负重、稳定值
Camera/Vision：进入观察模式
OutpostSystem：接收本局前哨点
ContainerSystem：启动刷新计时
SceneRandomEventDirector：写入本局事件结果
RunTimerController：使用本局 run_duration_seconds 初始化倒计时
MonsterSystem：接收怪物事件和抽中的点位
RandomObstacleSystem：接收障碍事件和抽中的点位
ExtractionController：撤离未解锁
```

## 8. 验收标准

| 编号 | 标准 |
|---|---|
| A1 | 启动新局后玩家出现在家安全区中心或指定入口 |
| A2 | 稳定值为 100，黑暗遮罩关闭，镜头为观察模式 |
| A3 | 家中临时存储为 4 格 |
| A4 | 第一前哨站从 3 个候选点中选 1 个 |
| A5 | 第二前哨站从 4 个候选点中选 1 个 |
| A6 | 撤离按钮未激活 |
| A7 | 候选点不足时不进入局内，并给出明确错误 |
| A8 | 第 3 日初始化时怪物事件激活并抽取 4 个怪物点位 |
| A9 | 超级时间事件能让本局倒计时初始化为 480 秒 |
| A10 | 超短时间事件能让本局倒计时初始化为 210 秒 |
| A11 | 随机障碍事件能抽取 2-4 个障碍点并完成路径校验 |

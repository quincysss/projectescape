# BDD 需求文档规范

> 目标：后续需求文档整理统一按 BDD（Behavior Driven Development）表达，让设计、实现和验收指向同一组可验证行为。
> 本文件约束“需求型文档”和“验收型文档”的写法，不替代系统母版和执行拆分文档。

## 0. 当前迁移状态

2026-05-15 已对旧需求、规则、验收、AI/MCP 提示词类文档完成第一轮初始迁移：补入统一的 `BDD 场景补充` 章节。

初始迁移只负责把旧文档接入 BDD 验收链路，不代表每个系统的成功、失败、边界场景已经全部细分完成。后续整理任一系统时，应继续把通用 Scenario 拆成更具体的业务 Scenario。

## 1. 适用范围

以下文档新增或重整时必须使用 BDD：

- 新系统需求文档。
- 系统功能规则细化文档。
- AI/MCP 开工提示词中的需求段落。
- 验收清单、最小交付清单、调试验证清单。
- 对玩家行为、系统状态、数据变化、UI 反馈有明确预期的规则说明。

以下内容可以不强制写成 BDD，但需要引用 BDD 场景：

- 设计母版中的体验目标和长期边界。
- 数据表字段说明。
- 资源规格、命名规则、路径规范。
- 工程结构、重构路线、技术债记录。

## 2. 基础格式

每个需求块优先使用以下结构：

```gherkin
Feature: 功能名称
  作为 <玩家/系统/开发者>
  我希望 <完成某个行为>
  以便 <产生明确价值>

  Background:
    Given <所有场景共享的前置状态>

  Scenario: 场景名称
    Given <前置条件>
    And <补充前置条件>
    When <触发行为>
    Then <可观察结果>
    And <数据/状态变化>
```

允许中文书写，但保留 `Feature / Background / Scenario / Given / When / Then / And / But` 关键词，方便 AI 和工具搜索。

## 3. 游戏系统写法

游戏需求场景必须尽量写清四类信息：

| 类型 | 要写清的问题 |
|---|---|
| 玩家行为 | 玩家做了什么、在哪里做、通过哪个 UI/交互触发 |
| 系统状态 | 当前天数、章节、背包/仓库/稳定值/任务状态等 |
| 数据变化 | 哪个字段、表格、存档、实例列表发生变化 |
| 可见反馈 | UI、音效、动画、提示、禁用态、错误提示如何表现 |

示例：

```gherkin
Feature: 制作所配方制作
  作为局外阶段玩家
  我希望消耗仓库材料制作已解锁配方
  以便把资源转化成可用于下一局的道具

  Scenario: 材料足够时制作成功
    Given 玩家已解锁 recipe_basic_filter
    And 局外仓库中有该配方所需材料
    And 仓库剩余容量可以容纳产物
    When 玩家点击制作按钮
    Then 系统从局外仓库扣除对应材料实例
    And 系统向局外仓库加入配方产物实例
    And 制作界面刷新材料数量和仓库容量
```

## 4. 场景组织规则

- 一个 `Feature` 对应一个功能点，不对应整个大系统。
- 一个 `Scenario` 只描述一个可验证行为。
- 前置条件写在 `Given`，触发动作写在 `When`，结果和状态变化写在 `Then`。
- 不把实现细节写成玩家需求；实现路径放在拆分执行文档的接口/模块段落。
- 数值和字段必须能追溯到表格、存档字段或系统文档。
- 失败、边界、空配置、容量不足、条件未满足等场景必须单独写。
- 自动触发和手动触发必须拆成不同场景。

## 5. 场景大纲

当同一行为有多组参数时，用 `Scenario Outline` 和 `Examples`：

```gherkin
Scenario Outline: 不同仓库容量研究档位生效
  Given 玩家完成 warehouse_capacity 的 <level> 档研究
  When 局外仓库刷新容量
  Then 仓库最大容量应为 <capacity>

Examples:
  | level | capacity |
  | Lv.1  | 80       |
  | Lv.2  | 90       |
  | Lv.3  | 100      |
  | Lv.4  | 110      |
  | Lv.5  | 120      |
```

## 6. 验收映射

每个 BDD 场景至少映射一种验证方式：

| 验证类型 | 写法 |
|---|---|
| 自动测试 | 标注测试文件、测试函数或预期新增测试 |
| Debug 验证 | 标注 GM/Debug 操作入口和观察项 |
| 手动验收 | 标注操作步骤和期望结果 |
| 文本验证 | 标注需要 `rg` 搜索的旧口径、字段或路径 |

推荐格式：

```text
验收方式：
- 自动测试：
- Debug 验证：
- 手动验收：
- 文本验证：
```

没有自动化测试时必须写明“未覆盖自动化测试”，并给出手动或 Debug 验证路径。

## 7. 文档维护要求

新增或修改需求文档后必须检查：

```text
是否有 Feature：
是否有至少一个 Scenario：
是否覆盖成功场景：
是否覆盖失败/边界场景：
是否写清 Given/When/Then：
是否映射验收方式：
是否需要同步系统 AI 开工入口：
是否需要同步系统文档索引表：
```

如果旧文档暂时无法完整 BDD 化，允许先新增“BDD 场景补充”章节，但后续整理时应逐步把关键需求迁移到 BDD 场景。

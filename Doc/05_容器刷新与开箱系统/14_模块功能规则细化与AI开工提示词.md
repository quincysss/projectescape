# 05 模块功能规则细化与 AI 开工提示词

> 来源文档：`05_容器刷新与开箱系统_修订版_废土生存法则.md`
> 用途：作为 05 模块的 AI/MCP 总入口。后续让 AI 在 Godot 中开工时，优先读取本文件，再按任务跳转到对应拆分文档。

## 1. 总体实现目标

容器刷新与开箱系统负责把地图探索转化为短周期奖励：

```text
刷新点合法性检查
-> 圈层与类型权重选择
-> 容器实体生成
-> 生命周期倒计时
-> 玩家长按开箱
-> 开启完成时生成战利品
-> 拾取进背包或保留在容器
-> 容器过期清理
```

系统必须服务废土生存节奏：玩家需要冒险离开安全区，找到临时机会，承担开箱时的暴露风险，再把资源带回循环中。

## 3. 推荐 Godot 模块边界

### 3.1 容器系统拥有

- 容器类型定义。
- 容器刷新点读取。
- 容器刷新节奏。
- 容器状态机。
- 开箱读条流程。
- 容器可交互状态输出。
- 战利品生成请求。
- 容器内剩余物管理。
- Debug 容器工具。

### 3.2 容器系统不拥有

- 背包容量最终判定。
- 玩家移动惩罚最终计算。
- 安全区数据维护。
- 地图圈层定义来源。
- 核心修复材料主产出设计。
- 多人网络权威同步。

## 4. 核心规则总表

| 项目 | V0.1 规则 |
| --- | --- |
| 刷新间隔 | 10-15 秒 |
| 单轮数量 | 1-3 个容器 |
| 点位要求 | 合法、空闲、未冷却 |
| 点位冷却 | 5-10 秒 |
| 容器状态 | SPAWNING / AVAILABLE / OPENING / OPENED / LOOTING / OPENED_AVAILABLE / EXPIRED / REMOVED |
| 开箱方式 | 靠近后长按 F |
| 中断条件 | 离开范围、移动、松开按键、稳定值耗尽 |
| 中断进度 | V0.1 重置 |
| 交互描边 | 统一 Shader 根据容器 Sprite alpha 自适应描边 |
| 掉落时机 | 开启完成时生成 |
| 掉落可预览 | 正式 UI 禁止，Debug 可预览 |
| 安全区 | 家园、前哨站、撤离点禁刷 |
| 背包满 | 未拾取物品留在容器 |
| 超重 | 不阻止拾取，只警告/惩罚 |
| 多人 | 只预留字段 |

## 7. 分阶段 AI 开工提示词

### 7.1 阶段一：配置与刷新点

```text
请读取 Doc/05_容器刷新与开箱系统/01_容器类型与配置表.md、
03_刷新点位与圈层权重.md、04_刷新管理器与轮次节奏.md。

在 Godot 中实现容器定义、刷新点数据、圈层权重和刷新管理器。
要求：
- 建立 7 类容器配置。
- 支持 4 个圈层的容器权重。
- 每 10-15 秒刷新一轮，每轮 1-3 个容器。
- 只在合法空点生成，点位有 5-10 秒冷却。
- 接入安全区查询接口，但如果安全区服务未完成，可以先做适配层。
- 提供 Debug 强制刷新。

请直接修改工程，并报告新增脚本、资源和验证步骤。
```

### 7.2 阶段二：容器状态机与开箱

```text
请读取 Doc/05_容器刷新与开箱系统/05_容器实体生命周期.md、
06_开箱交互与读条.md、07_开启锁定与过期规则.md。

实现容器实体状态机和玩家长按开箱。
要求：
- 实现 SPAWNING、AVAILABLE、OPENING、OPENED、LOOTING、OPENED_AVAILABLE、EXPIRED、REMOVED。
- 玩家必须在交互半径内长按 F 才能开箱。
- 离开范围、移动、松开按键、稳定值耗尽会中断。
- V0.1 中断后读条重置。
- 开箱期间容器倒计时继续。
- 若读条完成时容器刚好过期，仍允许打开结果结算。
- 已打开容器再次查看不需要读条。
- 可交互描边接入 InteractionHighlightService，使用 Shader 按 Sprite alpha 自动描边，不要为每种容器制作固定 PNG 边框。

请优先用项目现有输入、交互、UI 进度条模式。
```

### 7.3 阶段三：掉落与拾取

```text
请读取 Doc/05_容器刷新与开箱系统/08_战利品生成器.md、
09_拾取面板与背包负重接口.md。

实现开启完成时的战利品生成和拾取面板。
要求：
- 掉落只在开启完成时生成，不允许提前在正式 UI 暴露。
- 掉落结果由 container_type、ring_id、loot_seed、loot_table 决定。
- 支持 C/B/A/S 品质权重。
- 小箱子 1-2 或 1-3 个物品，大保险柜 2-4 个物品。
- 单个拾取和一键拾取都要支持。
- 背包满时未拾取物品留在容器中。
- 超重不阻止拾取，只触发警告或移动惩罚。

请保留 Debug 掉落预览，但不要接入正式玩家 UI。
```

### 7.4 阶段四：异常箱、安全区与 Debug

```text
请读取 Doc/05_容器刷新与开箱系统/10_异常预兆与高价值机会.md、
11_安全区负重多人预留边界.md、12_调试工具与验证清单.md。

补齐异常箱、安全区过滤和 Debug 面板。
要求：
- 异常箱只在最外层出现，权重 15。
- 可选实现 10 秒预兆和 30 秒异常箱窗口。
- 异常箱不保底 S 品质。
- 家园、已修复前哨站、修复交互点、撤离点禁止刷新容器。
- Debug 面板显示刷新倒计时、活跃容器、点位合法性、选中容器状态、seed。
- 提供强制刷新、清空、指定类型生成、异常箱生成、显示点位、预览掉落。

请跑一遍验证清单，并报告通过/失败项。
```

## 8. 总体验收标准

- 容器刷新闭环可以在 Godot 编辑器中运行。
- 随机刷新可通过 Debug 强制触发。
- 容器位置遵守圈层和安全区。
- 玩家可以长按打开容器。
- 容器进入交互半径时有贴合自身轮廓的 Shader 描边反馈。
- 中断、过期、再次查看逻辑正确。
- 战利品在开启完成时生成。
- 背包满和超重处理正确。
- 异常箱作为高价值机会出现，但不破坏品质概率。
- Debug 面板足以定位刷新、掉落、拾取问题。

## 9. 给后续 AI 的硬约束

```text
不要把本系统做成纯 UI 假流程。
不要删除原始 05 文档。
不要绕过背包系统直接改玩家物品。
不要让容器刷新进安全区。
不要让高价值容器稳定出现在家门口。
不要让核心修复材料依赖容器随机掉落。
不要在正式 UI 中提前显示未开启容器的掉落内容。
不要为不同容器分别制作或依赖固定 PNG 描边边框；交互描边优先走 Shader 和统一服务。
不要实现多人同步，只预留字段。
```

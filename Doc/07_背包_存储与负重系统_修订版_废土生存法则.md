# 《废土生存法则》背包、存储与负重系统（修订版）

> 文档版本：v0.2  
> 所属项目：《废土生存法则》  
> 前置文档：01_项目总纲.md / 02_核心循环规则.md / 03_地图与安全区规则.md / 04_稳定值与视野系统.md / 05_容器刷新与开箱系统.md / 06_前哨材料与前哨站修复系统.md  
> 适用阶段：V0.1 单人核心闭环 / Godot 原型开发  
> 文档目标：明确玩家背包、物品格子、堆叠、装备携带、家中存储、前哨站存储、局外仓库接口、负重计算、超重惩罚、丢弃销毁与 Godot 实现接口，让“带什么、拿多少、存哪里、丢不丢”成为清晰可执行的核心决策。

---

## 1. 本文档一句话说明

背包、存储与负重系统共同决定玩家能带走多少收益，以及带着这些收益还能不能活着回去。

它们服务的核心问题是：

```text
我还能拿吗？
我拿了会不会太重？
我应该先存起来吗？
我要不要丢掉低价值物？
```

---

## 2. 系统设计目标

## 2.1 核心目标

本系统需要达成：

```text
限制玩家无脑拾取。
让高价值和高重量形成取舍。
让家和前哨站存储有实际价值。
让死亡损失和撤离收益边界清楚。
让容器拾取、前哨材料、装备配置都能共用物品规则。
为局外仓库、研究所、制作所预留接口。
为局外商人出售和货币收益提供道具价值来源。
```

---

## 2.2 不做什么

V0.1 不建议做：

```text
俄罗斯方块式背包形状
地面掉落堆物品
复杂排序筛选
复杂拖拽动画
装备耐久
物品嵌套容器
背包内自由堆叠拆分
局内存储永久保留
```

核心原则：

```text
格子限制负责容量压力。
负重限制负责贪婪压力。
玩家必须能自主决定存入和取出哪些物品。
```

---

## 3. 物品基础规则

## 3.1 物品字段

所有可携带物品应具备：

```text
item_id
display_name
item_type
quality
quantity
stackable
stack_limit
weight_per_unit
total_weight
icon_id
sellable
sell_currency_id
sell_value
source
```

可选字段：

```text
equip_slot
consumable_type
blueprint_recipe_id
is_outpost_material
is_unique
```

注意：

```text
item_id 对应配置表中的静态物品定义。
item_instance_id 对应运行时生成的具体实例。
材料、装备、消耗品、图纸都应由配置表驱动，不建议硬编码在脚本里。
```

---

## 3.2 物品类型

V0.1 物品类型：

| 类型 | 说明 |
|---|---|
| 普通材料 | 局外研究、制作使用 |
| 前哨材料 | 局内修复前哨站使用 |
| 回复物品 | 回复稳定值或提供短时效果 |
| 装备 | 可装备，影响属性，也有重量 |
| 图纸 | 解锁制作配方 |
| 稀有物资 | 高价值结算或收藏 |

---

## 3.3 品质

品质：

```text
S
A
B
C
```

品质用于：

```text
结算展示
一键入库优先级
容器产出权重
物品价值预期
局外商人售价参考
```

品质不直接决定重量。

规则：

```text
低品质物品也可以很重。
高品质物品也可以很轻。
```

## 4. 背包格子规则

## 4.1 背包容量

V0.1 建议：

```text
玩家初始背包：6格
```

后续可通过：

```text
背包装备
研究所科技
特殊装备词条
```

提升容量。

---

## 4.2 格子坐标

背包格子从左上角开始计算。

建议使用：

```text
row, column
```

例如：

```text
(0,0) = 第一行第一格
(0,1) = 第一行第二格
(1,0) = 第二行第一格
```

注意：早期草稿中出现“往左为（0,1）”，正式规则建议统一为：

```text
向右 column +1
向下 row +1
```

这样更符合 UI 和数组习惯。

---

## 4.3 占格规则

V0.1 简化为：

```text
每个非堆叠物品占1格。
每种堆叠物品占1格。
堆叠数量不超过 stack_limit。
```

示例：

```text
废铁片 × 3，占1格。
旧电池 × 1，占1格。
滑板鞋 × 1，占1格。
图纸 × 1，占1格。
```

---

## 4.4 堆叠规则

可堆叠物品：

```text
普通材料
前哨材料
部分消耗品，可选
```

不可堆叠物品：

```text
装备
图纸
稀有收藏物，默认不可堆叠
```

V0.1 推荐：

```text
材料可堆叠。
消耗品暂不堆叠。
装备、图纸、稀有物资不堆叠。
```

---

## 4.5 拾取入包规则

拾取物品时：

```text
先尝试堆叠到已有同类格。
如果无法堆叠或堆叠满，则寻找空格。
如果无空格，则拾取失败。
拾取成功后重新计算负重。
```

如果拾取后超重：

```text
仍允许拾取。
显示负重提示。
应用速度惩罚。
```

负重不阻止拾取，背包格子阻止拾取。

---

## 5. 装备栏与消耗品栏

## 5.1 装备栏

V0.1 可先预留装备栏，不必做复杂装备系统。

建议装备槽：

```text
头部
身体
手部
腿部
背部
特殊
```

其中背部可对应背包装备，影响：

```text
背包容量
负重上限
移动修正
```

---

## 5.2 装备重量

装备也计入负重。

规则：

```text
已装备装备重量计入当前负重。
背包内未装备装备也计入当前负重。
装备替换后重新计算负重。
```

---

## 5.3 消耗品栏

V0.1 可先不做完整局外出发配置，但需要预留。

规则：

```text
初始消耗品栏：0格
玩家初始未解锁消耗品携带功能
完成研究所对应研究后解锁第1格
后续研究所最高提升到4格
对应按键：1、2、3、4
每格只能放1个消耗品
同类型消耗品只能携带1个
```

解锁后，消耗品也计入负重。

## 6. 负重系统

## 6.1 负重定义

当前负重：

```text
当前负重 = 背包物品重量 + 已装备装备重量 + 消耗品栏物品重量
```

物品总重量：

```text
total_weight = weight_per_unit × quantity
```

---

## 6.2 负重上限

V0.1 建议：

```text
基础负重上限：20
```

来源：

```text
玩家基础值
背包装备加成
研究所科技加成，后续
装备词条加成，后续
消耗品临时加成，后续
```

V0.1 最小实现：

```text
基础负重值
背包装备负重加成，可选
```

---

## 6.3 负重阶段

| 阶段 | 当前负重比例 | 状态 | 移动速度倍率 |
|---|---:|---|---:|
| 正常 | ≤100% | 正常 | 100% |
| 超重 | >100% 且 ≤150% | 负重过高 | 80% |
| 严重超重 | >150% | 严重超重 | 55% |

玩家基础移动速度暂定：

```text
6 单位/秒
```

---

## 6.4 超重规则

超重不阻止行动。

超重会：

```text
降低移动速度。
显示负重警告。
让角色移动表现变沉。
增加玩家返程压力。
```

不建议 V0.1 做：

```text
超重无法移动
超重掉稳定值更快
超重影响开箱速度
```

这些可以后续再评估。

---

## 6.5 负重提示

UI 需要显示：

```text
当前负重 / 负重上限
负重阶段颜色
超重提示
严重超重提示
```

建议表现：

| 阶段 | 表现 |
|---|---|
| 正常 | 白色或浅色 |
| 超重 | 粉红/橙色提示 |
| 严重超重 | 红色闪烁，角色喘息 |

## 7. 家中存储

## 7.1 家中容量

V0.1 建议：

```text
家中存储：4格
```

后续可通过研究所升级。

---

## 7.2 家中存储规则

玩家在家安全区内可以将背包物品存入家中存储。

规则：

```text
进入家安全区不会自动打开背包或家中存储界面。
家中撤离信号 UI 下方显示“按 TAB 打开背包界面”提示。
玩家在家中按 TAB 时，背包与家中存储双栏同时打开。
玩家离开家安全区时，如果家中背包/存储界面仍打开，则自动关闭。
存入后物品从背包移除。
当前负重重新计算。
家中存储物品死亡后保留进入失败结算。
撤离成功后进入成功结算。
本局结束后家中存储清空。
```

---

## 7.3 家中存储限制

V0.1 家中存储不等于局外仓库。

不能：

```text
直接访问局外仓库。
永久保存到下一局。
制作装备。
研究升级。
```

家中存储只是本局临时安全箱。

---

## 8. 前哨站存储

## 8.1 前哨容量

V0.1 建议：

```text
每个已修复前哨站：2格
```

---

## 8.2 前哨存储规则

玩家在已修复前哨站安全区内可以存取物品。

规则：

```text
未修复前哨站没有存储。
已修复前哨站开启2格存储。
存入后玩家负重重新计算。
前哨存储物品死亡后保留进入失败结算。
撤离成功后进入成功结算。
本局结束后前哨存储清空。
```

---

## 8.3 前哨存储设计目的

前哨存储不是仓库，而是：

```text
外层探索中转点
负重卸货点
死亡风险缓冲
继续贪容器前的安全选择
```

---

## 9. 容器残留与存储边界

05 文档定义：已开容器如果未拿空，可以保留到倒计时结束。

这里需要明确边界：

```text
容器残留不是存储。
玩家不能把背包物品放回容器。
容器倒计时不暂停。
容器倒计时归零后会消失。
容器内残留物不进入结算。
```

真正安全的局内存储只有：

```text
家中存储
已修复前哨站存储
```

---

## 10. 丢弃规则

## 10.1 丢弃方式

玩家打开背包后，可以丢弃物品。

V0.1 推荐交互：

```text
选择物品。
点击丢弃按钮。
出现确认，可选。
确认后物品销毁。
```

如果拖拽体验已经成熟，也可以：

```text
拖出背包区域丢弃。
```

---

## 10.2 丢弃结果

丢弃后：

```text
物品从背包移除。
物品直接销毁。
不会掉落到地图上。
不会进入任何容器。
不会进入结算。
当前负重重新计算。
```

---

## 10.3 为什么不掉落

V0.1 不做地面掉落。

原因：

```text
避免玩家把地面当临时仓库。
减少地图实体管理复杂度。
减少多人归属问题。
强化“丢弃就是损失”的决策。
```

---

## 11. 背包界面规则

## 11.1 打开条件

玩家按 TAB 打开背包。

建议：

```text
家中、前哨站、危险区都可以打开背包。
打开背包不暂停稳定值。
打开背包时玩家移动输入锁定。
关闭背包后恢复移动。
```

V0.1 单人也建议稳定值不暂停，为未来多人减少返工。

---

## 11.2 背包显示内容

背包界面显示：

```text
背包格子
物品图标
品质边框
堆叠数量
物品重量
当前负重 / 负重上限
装备栏，可选
消耗品栏，可选
当前可访问存储区
丢弃按钮
```

---

## 11.3 存储界面

在家或已修复前哨站打开背包时，应同时显示对应存储，但打开方式需要区分：

规则：

```text
在家：不自动打开背包/存储；只在撤离信号 UI 下方提示按 TAB 打开。
在家按 TAB：显示背包 + 家中存储。
离开家：如果背包/家中存储界面打开，则自动关闭。
在已修复前哨站：是否自动打开由前哨存储研究决定；研究未解锁时不自动打开。
在已修复且存储研究解锁的前哨站：进入安全区自动显示背包 + 该前哨站存储。
在危险区不显示存储。
```

V0.1 必须支持：

```text
玩家手动选择背包物品存入当前安全点存储。
玩家手动选择存储物品取回背包。
```

原因：

```text
玩家费力带回的物资，必须由玩家决定哪些进入安全箱。
系统自动规则只能作为辅助，不能替代玩家选择。
```

## 12. 物品转移规则

## 12.1 背包到存储

玩家在安全点内可将物品从背包转入存储。

规则：

```text
目标存储有空格或可堆叠时，允许转入。
转入成功后背包移除对应数量。
转入后重新计算负重。
```

---

## 12.2 存储到背包

玩家也可以从存储取回物品。

规则：

```text
背包有空格或可堆叠时，允许取回。
取回后增加玩家负重。
如果取回后超重，允许，但提示。
```

---

## 12.3 一键存入

V0.1 需要加入一键存入，但它只是快捷操作，不替代手动存取。

建议规则：

```text
优先存入稀有物资。
再存入材料，包括前哨材料和普通材料。
再存入图纸。
装备默认不自动存入，除非玩家选择。
```

如果存储空间不足：

```text
按优先级尝试存入。
同优先级按品质从高到低。
同品质按背包格子顺序。
未能存入的物品留在背包中，并提示存储空间不足。
```

一键存入后，玩家仍可手动调整：

```text
从存储取回不想存的物品。
手动存入系统没有自动存入的物品。
```

---

## 13. 死亡与撤离物品归属

## 13.1 死亡失败

死亡后进入失败结算。

保留：

```text
家中存储
已修复前哨站存储
```

丢失：

```text
玩家背包
玩家身上未安全结算的装备，V0.1按背包处理
容器内残留物
地图上未拾取物
```

---

## 13.2 撤离成功

撤离成功后进入成功结算。

带回：

```text
玩家背包
家中存储
已修复前哨站存储
```

不带回：

```text
容器内残留物
地图上未拾取物
未修复前哨站相关物
```

---

规则由 08/09 文档细化。

本系统只需要提供结算物品列表：

```text
result_items
lost_items
source_type
source_id
```

来源类型：

```text
BACKPACK
HOME_STORAGE
OUTPOST_STORAGE
LOST_BACKPACK
```

## 15. 排序与一键入库预留

结算界面一键入库需要用到排序。

建议优先级：

```text
品质高优先：S > A > B > C
同品质按结算列表顺序
结算列表顺序可按来源和格子序生成
```

格子序：

```text
row 从小到大。
column 从小到大。
```

## 17.2 ItemDefinition

定义物品静态数据：

```text
item_id
display_name
item_type
quality
stackable
stack_limit
weight_per_unit
icon_id
sellable
sell_currency_id
sell_value
equip_slot
```

后续版本建议从 `.tab` 配置表加载 ItemDefinition。

`.tab` 文件本质是制表符分隔的 TSV 文件，可以用电子表格维护。

推荐基础物品表：

```text
data/items.tab
```

建议字段：

| 字段 | 说明 |
|---|---|
| id | 物品唯一ID |
| item_type | 物品类型：material/outpost_material/equipment/consumable/blueprint/rare |
| name | 显示名 |
| quality | 品质 |
| stackable | 是否可堆叠 |
| stack_limit | 堆叠上限 |
| weight | 单个重量 |
| icon | 图标路径 |
| sellable | 是否可出售给局外商人 |
| sell_currency_id | 售卖获得的货币 ID，V0.1 默认 mine_coin |
| sell_value | 单个物品基础售价 |
| description | 描述文本 |
| tags | 标签，用于掉落、筛选、制作 |
| enabled_version | 启用版本 |

示例表头：

```text
id	item_type	name	quality	stackable	stack_limit	weight	icon	sellable	sell_currency_id	sell_value	description	tags	enabled_version
```

材料扩展表：

```text
data/materials.tab
```

建议字段：

| 字段 | 说明 |
|---|---|
| id | 材料ID，需要与 items.tab 的 id 对应 |
| material_category | 基础/电子/医疗/工具/稀有/前哨 |
| research_value | 研究用途权重，可选 |
| crafting_value | 制作用途权重，可选 |
| outpost_material | 是否前哨修复材料 |
| default_drop_weight | 默认掉落权重 |


## 21. 当前设计体检与建议

## 21.1 合理点

背包格子和负重双限制是合理的。

格子限制让玩家不能无限拿不同物品。

负重限制让玩家即使有格子，也要考虑返程速度。

前哨存储会因此变得非常有价值。

---

## 21.2 需要注意的点

### 负重不要过度惩罚

V0.1 不建议让超重影响稳定值下降。

否则玩家会同时被速度和视野双重惩罚，可能太难。

---

### 背包界面不要太重

危险区操作背包时稳定值继续下降。

如果 UI 太复杂，会让玩家觉得是在和界面战斗。

V0.1 应保持：

```text
清晰格子
快速丢弃
快速存入
明确重量
```

---

### 容器残留要和存储区分

已开容器可残留，但不是安全存储。

必须保持：

```text
容器倒计时继续。
容器不能放入玩家物品。
容器残留不进结算。
```

## 24. 当前规则覆写：所有道具不可堆叠

自 V0.1 当前实现起，所有可进入背包、家中存储、前哨相关交互、容器掉落与结算流程的道具都视为不可堆叠道具。

规则：
```text
所有道具 stackable = false。
所有道具 stack_limit = 1。
每个道具实例单独占用 1 个背包格或存储格。
即使同一 item_id 获得多件，也必须拆成多个 amount = 1 的独立条目。
负重判断按单件道具逐件计算，是否能放入背包由剩余格子和加入后的总负重共同决定。
容器、材料拾取点、家中存储与后续 UI 选择都不得把同名道具合并成一组。
```

设计原因：
```text
玩家后续需要点击选择具体道具放入背包或存储。
单件单格能避免“同一组堆叠物部分可拿、部分不可拿”的复杂边界。
负重、格子、存储决策都回到具体道具实例，便于 UI 表现和后续丢弃/转移规则。
```

---

## 25. 当前规则覆写：道具选择与转移必须按单件实例

自 V0.1 当前实现起，所有可交互物品列表都必须支持点击或选择具体道具实例。

规则：
```text
容器打开后，玩家可以点击容器列表中的某一件道具，系统只尝试将这一件道具放入背包。
材料交互打开后，玩家可以点击材料列表中的某一件材料，系统只尝试将这一件材料放入背包。
在家中，玩家可以点击背包中的某一件道具，系统只尝试将这一件道具存入家中存储。
在家中，玩家可以点击家中存储中的某一件道具，系统只尝试将这一件道具取回背包。
局外仓库、局外商人、研究所、制作所、出发准备等后续所有道具列表，都必须以 item index / item instance 为选择单位。
选择、转移、出售、消耗、丢弃都不得按 item_id 自动合并处理。
```

实现约束：
```text
底层转移接口必须先选择单件实例，再调用背包、存储或仓库的容量与负重规则。
当目标空间或负重不足时，只拒绝当前被选择的道具，不影响同列表中的其他道具。
UI 可以保留“一键拾取 / 一键入库”按钮，但一键逻辑也必须逐件调用同一套单件转移接口。
后续所有新 UI 面板不得绕过单件选择接口直接修改数组。
```

## 26. 本文档结论

背包、存储与负重系统决定了玩家搜集行为的重量感。

它应该让玩家在每次拿东西时都产生判断：

```text
这个东西值不值得占一格？
这个东西会不会太重？
我能不能先存到前哨站？
我要不要丢掉低价值材料？
我现在还能安全回家吗？
```

只要这些问题自然出现，游戏就会从“跑过去捡东西”变成真正的搜集撤离决策。

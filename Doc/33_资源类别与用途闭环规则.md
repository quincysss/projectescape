# 33_资源类别与用途闭环规则

> 文档版本：v0.1
> 所属项目：《废土生存法则》
> 关联文档：`05_容器刷新与开箱系统_修订版_废土生存法则.md`、`07_背包_存储与负重系统_修订版_废土生存法则.md`、`10_研究所与制作所系统_修订版_废土生存法则.md`、`29_局外杂货店经营与界面重构路线图.md`、`31_掉落物资与制造经济规划.md`。
> 文档目标：定义资源类别、保底用途和商品化层级，避免玩家夜出后带回一堆无法经营、无法研发、无法转化的无效物资。

## 1. 最高原则

```text
玩家每次夜出不一定拿到最优资源，但不能拿到完全无用的资源。
```

资源系统必须让玩家形成以下判断：

```text
我缺什么。
哪里更可能找到。
找不到目标时有什么次优解。
带回来后至少能怎么变成钱或能力。
```

## 2. 玩家可见资源类别

玩家在局内和局外优先理解“资源类别”，不是 SS/S/A/B/C。

| 类别 | 主要用途 | 典型出口 |
|---|---|---|
| 基础材料 | 修补、包装、低级生活货 | 基础商品、家具制作、低级装备 |
| 零件类 | 维修、工具、设备结构 | 维修货品、工作台、装备部件 |
| 组件类 | 电子、照明、信号 | 照明货品、情报设备、探索装备 |
| 食物药品类 | 生存、医疗、冷藏商品 | 医疗货品、消耗品、冷藏货 |
| 武器装备类 | 防身、夜出能力、高风险商品 | 装备、安保设备、高价订单 |
| 烟酒珠宝类 | 交易、奢侈需求、高利润 | 直接上架、礼品包装、特殊顾客 |

品质保留为后台参数：

```text
掉落权重
基础售价
结算展示
稀有惊喜
订单价值
```

品质不作为玩家收集方向的主语言。

## 3. 商品化层级

可售物资采用三层结构：

| 层级 | 规则 | 作用 |
|---|---|---|
| 基础商品 | 单资源或单类别即可制造 | 保证每天能营业 |
| 进阶商品 | 多类别组合 | 提高利润，制造规划 |
| 订单/高级商品 | 指定稀有材料、任务条件或任务条件 | 形成追求和高额回报 |

示例：

```text
废金属 x1 -> 废料五金包
废金属 + 布料 -> 修补工具包
废金属 + 电子组件 + 任务条件 -> 简易电器
废金属若干 -> 升级某类商品或制作店面设备
```

## 4. 资源保底出口

每个常规资源至少拥有一个保底出口。

优先级：

```text
1. 制造基础商品
2. 直接低价出售或包装出售
3. 拆解/转化为同类别通用资源
4. 投入家具、装备或消耗品制作
5. 留作进阶配方或订单
```

研究所不作为材料出口。永久研发只消耗矿币并检查前置条件。

## 5. 第一章资源池约束

第一章玩家可感知资源应控制在 8-12 个。

配置表中可以存在更多资源，但早期掉落池、需求榜和配方不要同时投放太多。

建议第一章资源结构：

| 类型 | 数量建议 |
|---|---:|
| 基础材料 | 3-4 |
| 零件/组件 | 3-4 |
| 食物药品 | 2-3 |
| 烟酒珠宝 | 1-2 |
| 武器装备/稀有件 | 少量事件或高风险地图投放 |

## 6. 设计禁区

不要做：

```text
早期配方大量要求指定材料。
研究所消耗局内随机材料。
材料只有品级没有用途类别。
常规资源只有一个窄用途。
玩家带回某类资源后完全无法卖、做、转化或保留。
```

## 7. 数据配置建议

### 7.1 resource_categories.tab

用途：定义玩家可见资源类别、背包堆叠口径、经营用途和地图推荐口径。

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| category_id | string | 类别 ID，如 basic / part / component |
| display_name | string | 玩家可见名称 |
| stack_policy | enum | material_stack / small_stack / instance_only |
| default_stack_limit | int | 默认堆叠上限 |
| primary_outlet | enum | sale/craft/equipment/fixture/order |
| source_hint | string | 常见来源短文本 |
| enabled_version | string | 启用版本 |

### 7.2 resource_outlets.tab

用途：定义每个常规资源的保底出口，避免“捡回来没有用”。

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| item_id | string | 资源物品 ID |
| category_id | string | 所属资源类别 |
| outlet_type | enum | basic_sale/direct_sale/convert/craft_fixture/craft_equipment/order |
| target_id | string | 对应商品、配方、转化或订单 ID |
| min_chapter | int | 最早出现章节 |
| priority | int | 同资源多个出口的展示优先级 |
| enabled_version | string | 启用版本 |

### 7.3 sale_good_recipes.tab 的类别需求

基础可售商品必须支持“按类别消耗”，不要全部绑定具体材料。

示例：

```text
医疗应急包：food_medical x1 + basic x1
五金杂货包：part x1
简易礼品包：luxury x1 或 direct_sale
```

进阶商品和订单可以要求具体材料，但第一章不应让指定材料成为日常营业的主要卡点。

## 8. Godot 模块建议

建议拆出以下服务：

| 模块 | 职责 |
|---|---|
| ResourceCategoryService | 查询资源类别、堆叠策略、常见来源 |
| ResourceOutletValidator | 启动时校验常规资源是否存在保底出口 |
| SaleGoodRecipeMatcher | 按库存类别匹配基础商品和组合商品 |
| ResourceConversionService | 处理同类别转化、拆解和低价出售 |

这些服务不负责生成掉落。掉落仍由容器/地图系统负责，资源服务只回答“这个东西拿回来能做什么”。

## 9. 校验规则

版本启用时必须检查：

```text
所有 enabled 的常规 material 必须有 material_category。
所有 enabled 的常规 material 至少有一个 resource_outlets 记录。
第一章同时投放的可见资源控制在 8-12 个。
基础商品配方至少覆盖 basic、part、component、food_medical 中的主要类别。
研究表不得引用具体材料作为消耗。
```

高稀有订单物、剧情物和唯一物可以没有基础出口，但必须标记为 quest / order / unique，不能混入常规资源池。

## 10. BDD 场景补充

Feature: 资源类别与用途闭环

  Scenario: 玩家带回任意常规资源后都有保底出口
    Given 玩家从夜间探索带回一个常规 material
    When 玩家进入白天准备阶段
    Then 系统应至少提供一种基础商品、低价出售、同类转化、家具制作或装备消耗出口

  Scenario: 研发不再被随机材料卡住
    Given 玩家拥有足够矿币
    And 研究前置条件已经满足
    When 玩家点击研究
    Then 系统只扣除矿币
    And 不检查或扣除具体材料

  Scenario: 基础商品保证可营业
    Given 玩家只带回单一类别的基础资源
    When 玩家打开制造所
    Then 系统应提供至少一种低利润可售物资制造路径

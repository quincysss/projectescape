# 06-05 前哨站状态机

> 来源文档：`06_前哨材料与前哨站修复系统_修订版_废土生存法则.md`
> 施工补充：开工前必须读取 `14_模块功能规则细化与AI开工提示词.md`。

## 1. 实现目标

为前哨站建立清晰状态机，统一候选、生成、待修复、修复中、已激活和失效等流程。

## 2. 状态建议

```gdscript
enum OutpostState {
    CANDIDATE,
    LOCKED,
    DAMAGED,
    REPAIRABLE,
    REPAIRING,
    ACTIVE,
    DISABLED
}
```

状态含义：

| 状态 | 说明 |
| --- | --- |
| CANDIDATE | 地图候选点，尚未被本局选中 |
| LOCKED | 已选中但还未暴露或初始化完成 |
| DAMAGED | 已出现，可查看需求但不可修复完成 |
| REPAIRABLE | 材料满足，可开始修复 |
| REPAIRING | 玩家正在读条修复 |
| ACTIVE | 修复完成，安全区/存储/撤离联动生效 |
| DISABLED | 局结束或特殊规则禁用 |

## 3. 转换规则

```text
CANDIDATE -> LOCKED -> DAMAGED
DAMAGED -> REPAIRABLE：需求满足
REPAIRABLE -> REPAIRING：玩家开始修复
REPAIRING -> REPAIRABLE：修复中断
REPAIRING -> ACTIVE：读条完成
ACTIVE -> DISABLED：局结束或清理
```

## 4. 状态职责

`OutpostEntity` 负责：

- 当前状态。
- 需求进度。
- 可交互提示。
- 修复读条。
- 激活后组件启用。

`OutpostManager` 负责：

- 创建前哨站。
- 保存前哨站列表。
- 广播激活状态。
- 与撤离/安全区/存储系统对接。

## 5. 信号

```gdscript
signal outpost_state_changed(outpost_id: String, old_state: int, new_state: int)
signal outpost_became_repairable(outpost_id: String)
signal outpost_activated(outpost_id: String)
signal outpost_disabled(outpost_id: String)
```

## 6. AI/MCP 开工提示词

```text
请实现前哨站状态机。

要求：
1. 状态包含 CANDIDATE、LOCKED、DAMAGED、REPAIRABLE、REPAIRING、ACTIVE、DISABLED。
2. 材料需求满足时从 DAMAGED 进入 REPAIRABLE。
3. 玩家开始修复时进入 REPAIRING，中断回到 REPAIRABLE。
4. 读条完成进入 ACTIVE，并发出 outpost_activated。
5. ACTIVE 后启用安全区、临时存储和撤离联动。
6. 所有状态转换写 Debug 日志。
```

## 7. 验收标准

- 状态转换路径可追踪。
- 不满足材料时不能进入修复读条。
- 中断后不会错误激活。
- ACTIVE 只触发一次。

## 8. 暂不实现

- 前哨站被敌人破坏。
- ACTIVE 后重新退化。

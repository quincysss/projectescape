# 06 模块功能规则细化与 AI 开工提示词

> 来源文档：`06_前哨材料与前哨站修复系统_修订版_废土生存法则.md`
> 用途：作为 06 模块的 AI/MCP 总入口。

## 1. 总体实现目标

前哨站系统要把地图探索变成局内中继建设：

```text
候选点随机
-> 前哨站出现
-> 生成修复需求
-> 材料在风险区域刷新
-> 玩家搬运并提交
-> 修复读条
-> 激活安全区/临时存储/撤离联动
```

核心体验是“为了更深入探索，玩家需要在风险中建立临时支点”。

## 3. 核心规则

- V0.1 推荐 2 个前哨站阶段。
- 前哨站候选点每局随机选择。
- 候选点要有互斥规则，避免路线过近。
- 前哨材料主来源是专用材料点，不是容器随机掉落。
- 材料可拾取、占背包、可提交。
- 90 秒保底避免关键材料卡死。
- 材料满足后才能修复。
- 修复读条可中断，中断后进度重置。
- 修复完成后前哨站 ACTIVE。
- ACTIVE 后启用安全区、临时存储和撤离联动。
- 前哨存储是局内临时存储，不是永久仓库。
- 左侧目标链 HUD 必须显示“当前目标：收集前哨站材料（菱形） n/N -> 下一步：前哨站修复 -> 解锁撤离”，并随拾取、提交、修复、撤离解锁实时刷新。

## 5. 分阶段 AI 开工提示词

### 阶段一：候选点与前哨站生成

```text
请读取 06 拆分文档中的 01、02、05、11。
实现 OutpostDefinition、OutpostManager 和前哨站候选点随机。
V0.1 支持两个前哨站阶段，候选点按 stage_group 读取，过滤非法点并应用互斥组。
生成前哨站实体后进入 DAMAGED 状态。
请输出脚本、资源结构和验证步骤。
```

### 阶段二：材料需求与刷新

```text
请读取 06 拆分文档中的 03、04。
实现 OutpostMaterialDefinition、固定需求表、材料点位刷新和 90 秒保底。
材料主来源必须是地图材料点，不要依赖容器随机掉落。
拾取材料后更新需求提示，但不自动提交。
```

### 阶段三：提交、修复与激活

```text
请读取 06 拆分文档中的 05、06、07、08。
实现材料提交 UI、修复读条、中断规则、ACTIVE 激活、安全区注册和前哨临时存储。
存储转移必须调用 07 背包/存储接口，不要复制物品系统。
```

### 阶段四：撤离结算与 Debug

```text
请读取 06 拆分文档中的 09、10、12、13。
实现撤离条件接口、死亡/撤离结算 payload、多人预留字段和 Debug 面板。
Debug 面板要能强制选点、生成材料、提交材料、完成修复并显示状态。
```

### 阶段五：左侧目标链 HUD

```text
请读取 06 拆分文档中的 15，并联动 08_撤离与结算系统/02_撤离条件与激活显示.md。
将左侧金黄色前哨信息框改造成目标链 HUD：
标题显示“当前目标：收集前哨站材料（菱形） n/N”，第二行显示“下一步：前哨站修复”，第三行显示“解锁撤离：未解锁/已解锁/可撤离”。
HUD 只读取 OutpostManager 和 ExtractionController 的视图模型，不直接计算业务状态。
拾取菱形材料、提交材料、修复进度变化、前哨 ACTIVE、撤离解锁时必须刷新。
```

## 6. 总体验收标准

- 每局前哨站候选点可变化。
- 前哨材料不会随机卡死。
- 玩家能提交材料并修复。
- 修复中断规则有效。
- ACTIVE 后安全区、存储、撤离联动生效。
- 死亡失败不会把前哨收益永久带回。
- Debug 面板能跑完整闭环。
- 左侧目标链 HUD 能让玩家理解菱形材料、前哨修复和撤离解锁的因果链。

## 7. 给后续 AI 的硬约束

```text
不要删除原始 06 文档。
不要把前哨站做成永久家园。
不要把前哨材料主来源做成容器随机掉落。
不要绕过 07 背包系统实现独立存储。
不要让材料过重到破坏探索。
不要把左侧黄色框继续做成只显示“I/II 未修复”的状态列表。
不要实现多人同步，只预留字段。
```

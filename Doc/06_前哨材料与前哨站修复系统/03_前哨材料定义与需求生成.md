# 06-03 前哨材料定义与需求生成

> 来源文档：`06_前哨材料与前哨站修复系统_修订版_废土生存法则.md`
> 施工补充：开工前必须读取 `14_模块功能规则细化与AI开工提示词.md`。

## 1. 实现目标

把前哨站修复材料做成明确的任务材料：它们可以占背包、可拾取、可提交，但不能主要依赖容器随机掉落。

## 2. 材料定位

前哨材料应满足：

- 用于修复前哨站。
- 是局内目标物。
- 主要由专用材料点刷新。
- 可以在地图上通过提示识别。
- 重量适中，促使玩家做路线选择。

不应满足：

- 变成普通容器掉落垃圾。
- 完全随机不可控。
- 单个材料过重导致玩家无法探索。

## 3. 物品属性建议

```gdscript
class_name OutpostMaterialDefinition
extends Resource

@export var id: String
@export var display_name: String
@export var weight: float
@export var stack_limit: int = 1
@export var stage_tags: Array[String]
@export var rarity: String
@export var icon: Texture2D
@export var world_scene: PackedScene
```

建议字段：

| 字段 | 规则 |
| --- | --- |
| `is_outpost_material` | 必须为 true |
| `target_stage` | 可用于第几个前哨站 |
| `submit_value` | 提交后满足多少需求 |
| `can_drop` | V0.1 可为 false |
| `can_store_home` | 可存家中，但局内需求主要现场解决 |

## 4. 需求生成规则

V0.1 可以使用固定需求，后续再随机。

第一前哨站建议：

```text
2 个基础维修材料
1 个电力/信号材料
```

第二前哨站建议：

```text
2-3 个结构材料
1-2 个电力/信号材料
1 个高风险区域材料
```

需求结构：

```gdscript
{
    "material_id": "cable_bundle",
    "required_count": 2,
    "submitted_count": 0
}
```

## 5. 显示规则

前哨站交互 UI 必须显示：

- 所需材料。
- 已提交数量。
- 背包中可提交数量。
- 缺少材料的来源提示。

材料世界物提示：

- 名称。
- 前哨材料图标。
- 重量。
- 目标前哨阶段或用途。

## 6. AI/MCP 开工提示词

```text
请实现前哨材料定义与需求生成。

要求：
1. 建立 OutpostMaterialDefinition 资源。
2. 前哨材料必须可被背包系统识别为 outpost_material。
3. V0.1 先使用两个阶段的固定需求表。
4. 需求结构包含 material_id、required_count、submitted_count。
5. 前哨站交互 UI 能显示需求、已提交、背包可提交数量。
6. 不要把前哨材料主来源做成容器随机掉落。
```

## 7. 验收标准

- 每个前哨站能生成明确需求。
- 背包中材料能被识别为可提交。
- UI 能显示缺少哪些材料。
- 材料重量不会严重压垮背包体验。

## 8. 暂不实现

- 复杂随机需求池。
- 材料品质影响修复。
- 材料替代配方。

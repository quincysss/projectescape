# 06-10 多人预留与权威边界

> 来源文档：`06_前哨材料与前哨站修复系统_修订版_废土生存法则.md`
> 施工补充：开工前必须读取 `14_模块功能规则细化与AI开工提示词.md`。

## 1. 实现目标

V0.1 不做多人，但前哨站修复、材料提交、存储和激活都要预留权威字段，避免后续多人重构。

## 2. 需要预留的字段

```gdscript
var repaired_by_player_id: String = ""
var current_repair_player_id: String = ""
var submitted_by_player_ids: Dictionary = {}
var storage_access_policy: String = "local_player"
var revision: int = 0
```

## 3. 权威边界

未来多人时应由服务器/主机权威决定：

- 候选点选择。
- 材料刷新。
- 材料提交。
- 修复完成。
- 前哨站 ACTIVE 状态。
- 存储物品列表。
- 撤离条件解锁。

V0.1 单机可以本地执行，但接口命名不要写死成单机。

## 4. 冲突预留

需要预留的冲突场景：

- 两名玩家同时提交同一类材料。
- 两名玩家同时开始修复。
- 玩家修复中断后另一名玩家接手。
- 多人同时打开前哨存储。

V0.1 可以通过 `local_player` 占位。

## 5. AI/MCP 开工提示词

```text
请为前哨站系统预留多人字段和权威边界，但不要实现网络同步。

要求：
1. OutpostEntity 包含 repaired_by_player_id、current_repair_player_id、submitted_by_player_ids、revision。
2. 材料提交和修复完成接口都接收 player_id。
3. V0.1 使用 local_player。
4. 状态改变时 revision +1。
5. 存储访问策略保留字段，但不做多人权限 UI。
6. 注释说明未来服务器/主机应拥有候选点、材料刷新、提交、修复、存储和撤离条件的权威。
```

## 6. 验收标准

- 接口包含 player_id。
- 状态变化会更新 revision。
- 单机流程不受多人字段影响。
- 后续可基于字段扩展同步。

# 工程结构重构路线图

> 目标：把 `BaseScene` / `RunScene` 从“大型业务脚本”逐步收敛成场景装配器，运行时规则放回对应模块目录，测试脚本只保留验证入口。

## 1. 当前结论

项目文档已经给出主模块边界，当前工程也已经有不少模块目录：

```text
scripts/base
scripts/run
scripts/inventory
scripts/outpost
scripts/events
scripts/monsters
scripts/map
scripts/profile
scripts/dialogue
scripts/ui
scripts/debug
```

当前主要问题不是“完全没有目录”，而是：

- `scripts/base/base_scene.gd` 同时承担局外 UI、货台/经营面板、研究面板、章节弹窗、Tooltip、Debug/GM 操作、夜晚出发 loading 和剧情触发。
- `scripts/run/run_scene.gd` 同时承担局内场景搭建、剧情视频、Web 视频桥、交互入口、容器/前哨/撤离胶水、存储 UI、镜头适配和世界 UI / 局内可读表现刷新。
- `tests/` 下目前主要是回归检查脚本，不应放运行时代码；后续新增运行时代码必须进 `scripts/<module>/`，测试只调用公开接口或加载场景验证。

## 2. 路径规则

新增或移动脚本时按职责归档：

```text
scripts/base/          局外基地装配、昼夜阶段协调、局外面板控制器
scripts/run/           单局运行编排、交互控制器、容器/材料刷新控制器
scripts/events/        场景随机事件解析、事件上下文、时间类事件应用
scripts/monsters/      怪物生成、行为、视野表现
scripts/outpost/       前哨点、前哨状态、前哨存储、前哨视觉锚点
scripts/inventory/     背包、仓库格子、转移、负重
scripts/profile/       玩家档案、保存适配器
scripts/dialogue/      对白读取、说话人、立绘控制
scripts/ui/            纯 UI 组件、Tooltip、结算、loading、通用样式
scripts/debug/         GM 调试服务、验证工具、调试面板动作
tests/                 只放自动化检查脚本，不放运行时模块
```

## 5. 需要设计确认的模块

## 6. 每轮重构约束

- 每轮只抽一个职责簇，避免大范围改名和移动导致场景引用断裂。
- 改前先跑对应测试，改后至少跑同一测试。
- 移动运行时代码时同步检查 `.tscn` 脚本引用。
- `tests/` 下新增脚本必须是验证脚本，不能作为运行时依赖。
- 如果某个模块需要新的设计边界，先记录到本文档，再进入实现。

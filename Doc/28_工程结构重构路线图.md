# 工程结构重构路线图

> 目标：把 `BaseScene` / `RunScene` 从“大型业务脚本”逐步收敛成场景装配器，运行时规则放回对应模块目录，测试脚本只保留验证入口。

## 1. 当前结论

项目文档已经给出主模块边界，当前工程也已经有不少模块目录：

```text
scripts/base
scripts/run
scripts/inventory
scripts/outpost
scripts/events
scripts/monsters
scripts/map
scripts/profile
scripts/dialogue
scripts/ui
scripts/debug
```

当前主要问题不是“完全没有目录”，而是：

- `scripts/base/base_scene.gd` 同时承担局外 UI、商人面板、研究面板、章节弹窗、Tooltip、Debug/GM 操作、出发 loading 和剧情触发。
- `scripts/run/run_scene.gd` 同时承担局内场景搭建、剧情视频、Web 视频桥、交互入口、容器/前哨/撤离胶水、存储 UI、镜头适配和世界 UI 刷新。
- `tests/` 下目前主要是回归检查脚本，不应放运行时代码；后续新增运行时代码必须进 `scripts/<module>/`，测试只调用公开接口或加载场景验证。

## 2. 路径规则

新增或移动脚本时按职责归档：

```text
scripts/base/          局外基地装配、页签协调、局外面板控制器
scripts/run/           单局运行编排、交互控制器、容器/材料刷新控制器
scripts/events/        场景随机事件解析、事件上下文、时间类事件应用
scripts/monsters/      怪物生成、行为、视野表现
scripts/outpost/       前哨点、前哨状态、前哨存储、前哨视觉锚点
scripts/inventory/     背包、仓库格子、转移、负重
scripts/profile/       玩家档案、保存适配器
scripts/dialogue/      对白读取、说话人、立绘控制
scripts/ui/            纯 UI 组件、Tooltip、结算、loading、通用样式
scripts/debug/         GM 调试服务、验证工具、调试面板动作
tests/                 只放自动化检查脚本，不放运行时模块
```

## 3. 已执行轮次

### 2026-05-14 第一轮：局外 Debug 动作服务

新增：

```text
scripts/debug/base_debug_action_service.gd
```

调整：

- `base_scene.gd` 的 Debug 按钮回调只负责调用服务、展示 message 和刷新页签。
- 补研究材料、补可售物、强制怪物、重置档案/剧情、推进天数、强制完成章节等规则逻辑从 `BaseScene` 移出。
- `base_scene.gd` 中用于局外 UI 查表的注册表改名为 `base_data_registry`，不再混用 `_debug_registry`。

## 4. 建议后续轮次

### 第二轮：局外商人/研究/仓库面板拆分

目标：

- `BaseScene` 只协调页签和 GameState。
- 商人购买/出售 UI 移到 `scripts/base/base_merchant_panel_controller.gd` 或 `scripts/ui/merchant_panel_controller.gd`。
- 研究树和材料需求格子移到 `scripts/base/base_research_panel_controller.gd`。
- 仓库格子渲染和 Tooltip 绑定移到 `scripts/ui/item_grid_view.gd`、`scripts/ui/item_tooltip_view.gd`。

优先级高，因为 `base_scene.gd` 的 UI 代码现在最多。

### 第三轮：局内剧情门控与视频桥拆分

目标：

- 第二日黑潮剧情、暂停 token、Web 视频桥从 `RunScene` 移到：

```text
scripts/flow/run_story_gate_controller.gd
scripts/ui/web_video_bridge.gd
```

需要注意：文档里已经提到 `RunSimulationPauseService`，但当前还没有独立服务。这个模块边界建议先确认。

### 第四轮：局内交互胶水拆分

目标：

- 容器打开、物品拾取、材料拾取、前哨修复、撤离长按都走各自控制器。
- `RunScene` 只负责把节点、控制器和信号连起来。

候选模块：

```text
scripts/run/container_interaction_controller.gd
scripts/outpost/outpost_repair_flow_controller.gd
scripts/extraction/extraction_flow_controller.gd
scripts/inventory/run_inventory_panel_controller.gd
```

### 第五轮：表现模块拆分

目标：

- 世界 UI 缩放、容器/前哨气泡、可读性字号、图标排版从玩法脚本中移出。
- 当前 `interactable_visual_builder.gd` 已经承担一部分表现职责，可以继续扩成明确的表现层。

候选模块：

```text
scripts/ui/world_readability_scaler.gd
scripts/ui/outpost_requirement_bubble_view.gd
scripts/ui/container_lifetime_view.gd
```

## 5. 需要设计确认的模块

以下边界文档里已有方向，但当前实现还缺少明确工程接口，建议后续由策划/程序确认后再拆：

- 任务系统：左侧目标链、章节目标、局内前哨目标是否统一为 `ObjectiveService`。
- 表现模块：世界 UI、HUD、特效、描边和音频反馈是否统一进入 `presentation` / `ui` 分层。
- Command 系统：文档建议关键行为走 Command，但当前大量 UI 仍直接调 GameState 或 RunScene 方法。
- RunSimulationPauseService：剧情、弹窗、结算、loading 是否统一使用 pause token。
- SceneRouter / GameFlowController：当前场景切换分散在 BaseScene、RunScene、LoadingScreen 中。

## 6. 每轮重构约束

- 每轮只抽一个职责簇，避免大范围改名和移动导致场景引用断裂。
- 改前先跑对应测试，改后至少跑同一测试。
- 移动运行时代码时同步检查 `.tscn` 脚本引用。
- `tests/` 下新增脚本必须是验证脚本，不能作为运行时依赖。
- 如果某个模块需要新的设计边界，先记录到本文档，再进入实现。

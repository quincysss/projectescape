# 00_落地执行文档总则

> 适用范围：`Doc/02_核心循环规则` 到 `Doc/10_研究所与制作所系统` 的拆分文档、V0.1/V0.2 清单、AI/MCP 开工提示词，以及后续新增需求型系统目录。  
> 目标：把设计文档转成可以被其他线程直接执行、验证、汇报的制作指令。

---

## 1. 文档定位

项目文档分为两类：

| 类型 | 用途 | 写法 |
|---|---|---|
| 设计母版 | 说明玩法意图、体验目标、系统边界 | 可以解释原因，可以保留概念描述 |
| 落地执行文档 | 指导线程实际创建场景、脚本、资源、配置、测试 | 必须像施工单，写清路径、操作、验收、禁止事项 |
| BDD 场景补充 | 把需求、规则和验收转成可验证行为 | 必须包含 `Feature`、`Scenario`、`Given`、`When`、`Then` |

拆分后的文档默认属于落地执行文档。

自 BDD 规范生效后，新增或重整需求、规则、验收、AI 开工提示词时，必须读取 `Doc/00_BDD需求文档规范.md`，并维护本文档末尾或对应章节中的 `BDD 场景补充`。

落地执行文档不应只说“要实现某系统”，而要回答：

```text
从哪个场景进入？
改哪些文件？
新增哪些节点？
脚本挂在哪？
字段叫什么？
由谁调用？
怎么验证？
失败时怎么汇报？
哪些内容禁止顺手做？
```

## 2. V0.1 固定入口

V0.1 阶段所有落地线程默认使用以下入口：

| 用途 | 固定路径 | 规则 |
|---|---|---|
| 正式初始场景 | `res://scenes/boot/BootScene.tscn` | 必须配置为 `project.godot` 的 `application/run/main_scene` |
| 局外基地 | `res://scenes/base/BaseScene.tscn` | BootScene 初期可直接进入 BaseScene |
| 局内场景 | `res://scenes/run/RunScene.tscn` | BaseScene 的出发入口进入 RunScene |
| 核心循环调试场景 | `res://scenes/debug/DebugCoreLoop.tscn` | 只用于开发验证，不作为正式 Main Scene |

任何玩法模块开工前，先检查这些入口是否存在。入口不清时，优先修入口，不继续堆玩法代码。

## 3. 单份落地文档必须包含

每份拆分文档至少包含以下内容：

```text
1. 实现目标
2. 设计依据
3. Godot 落地范围
4. 节点/脚本/资源路径
5. 字段与数据结构
6. 运行流程
7. 交互与信号接口
8. 开工前检查
9. 执行步骤
10. 验收标准
11. 验证命令或验证场景
12. BDD 场景补充
13. 暂不实现
14. 完成后汇报格式
```

如果某一项暂时无法确定，必须写：

```text
待定原因：
临时执行方案：
需要谁补充：
```

不要留空，也不要只写“后续完善”。

## 4. 开工提示词标准格式

每个 AI/MCP 开工提示词必须可以直接复制给执行线程，格式如下：

```text
任务：
在 Godot 中实现/修订 XXX 模块的 V0.1 最小闭环。

必须先读取：
- Doc/00_落地执行文档总则.md
- Doc/XX_系统名/00_拆分规范与落地文档写法.md
- Doc/XX_系统名/YY_具体模块.md
- 相关依赖文档

允许修改：
- res://scenes/...
- res://scripts/...
- res://data 或 res://resources/...
- tests/...

禁止修改：
- 与本模块无关的系统
- 设计母版文档
- 非必要资源大文件

执行步骤：
1. 检查入口场景和依赖节点。
2. 创建或修改脚本/场景/资源。
3. 接入现有信号或接口。
4. 增加最小验证。
5. 运行验证并记录结果。

完成标准：
- ...

最终报告：
- 修改文件
- 新增节点/脚本/资源
- 已验证内容
- 未实现内容
- 风险和后续
```

## 5. 交付报告格式

每个落地线程完成后，必须按以下格式汇报，方便文档线程反向维护：

```text
完成模块：

读取文档：

新增文件：

修改文件：

入口场景：

新增节点：

新增脚本/资源：

关键接口：

验证命令：

验证结果：

未实现范围：

发现的文档缺口：

建议文档修订：
```

如果没有运行验证，必须写明原因，不允许只写“未测试”。

---

## 6. 文档维护规则

其他线程落地后，如果发现文档与工程不一致，文档线程按以下优先级处理：

```text
1. 工程已明确采用且能运行的路径，反向写入落地文档。
2. 工程临时绕过但设计仍正确的规则，文档保留并标记当前未实现。
3. 工程实现证明规则不可行时，先写风险，再决定是否修订母版。
4. 不因为一次临时实现就修改长期设计目标。
```

---

## 7. 禁止事项

```text
不要把母版设计文档改成流水账。
不要在拆分文档里只复述玩法愿景。
不要写没有路径、没有操作、没有验收的“空提示词”。
不要让 Debug 场景替代正式入口。
不要让 UI 直接修改核心数据。
不要为了一个模块顺手实现其他模块的大范围功能。
```

## 8. 当前优先级

V0.1 文档维护优先级：

```text
1. 启动入口和场景路径。
2. 02 核心循环最小闭环。
3. 03 地图与安全区可玩空间。
4. 04 稳定值与视野死亡。
5. 07 背包/存储/负重底座。
6. 05 容器开箱。
7. 06 前哨材料与修复。
8. 08 撤离与结算。
9. 09 局外仓库。
```

后续文档可以继续扩展 V0.2，但不得阻塞 V0.1 可玩闭环落地。
## BDD 场景补充

> 初始迁移：本节把本文规则挂接到 BDD 验收链路。后续重整本文时，应继续把关键成功、失败和边界规则拆成独立 Scenario。

```gherkin
Feature: 00_落地执行文档总则
  作为 系统设计与实现线程
  我希望按本文规则完成 00_落地执行文档总则
  以便对应功能、数据和验收保持一致

  Scenario: 按本文规则完成核心行为
    Given 已读取 `Doc/00_AI开工入口.md`
    And 已读取 `Doc/00_BDD需求文档规范.md`
    And 当前任务属于 "Doc 顶层专题"
    When 根据本文新增、修改或验证对应功能
    Then 必须覆盖本文定义的前置条件、触发行为、结果反馈和边界情况
    And 必须把验证结果记录到本文验收标准或对应调试验证清单
```

验收方式：
- 文本验证：检查本文是否保留 `Feature`、`Scenario`、`Given`、`When`、`Then`。
- 执行验证：按本文既有“验收标准/调试工具/最小交付清单”执行。
- 缺口记录：若本文尚未拆出具体失败或边界场景，后续需求整理时继续补齐。
